# 第7回: 評価（Evaluation）とオブザーバビリティ

> システムを「作りっぱなし」にせず、3,000人の利用に耐えうる品質を維持・向上させるための「評価」と「観測可能性」。改善をデータ駆動にする手法。

---

## 1. Genkit Evaluators による「RAGの三要素」評価

商用レベルでは **RAG Triad（RAGの三要素）** を定量的（0.0〜1.0）に測定する。

* **Faithfulness（忠実性）**: 回答に嘘（ハルシネーション）がないか。
* **Answer Relevancy（回答の関連性）**: ユーザーの意図に沿っているか。
* **Context Precision（検索精度）**: リランキング後の上位結果に正解が含まれているか。

**実装のポイント**:
Genkitの [`evaluators`](https://firebase.google.com/docs/genkit/evaluation) を使い、Gemini 2.5 Pro等を「試験官」として稼働させる。開発時に設定（チャンクサイズ等）を変更するたびに、事前に用意した **「ゴールデン・データセット（100件の理想的なQ&Aペア）」** に対して自動テストを走らせ、スコアが下がった場合はデプロイをブロックするCI/CDパイプラインを構築する。

## 2. Genkit Trace による「推論プロセスの解剖」

3,000人規模で運用していると、「なぜかこの質問だけ答えがズレる」という特異なケースが必ず発生する。

* **トレースの記録内容**:
    Cloud Functions上での処理を Genkit の **Trace機能** で記録する。
    * どの検索クエリが生成されたか？
    * ベクトル検索で何位の資料がヒットしたか？
    * リランキング後のスコアは何点だったか？
* **Cloud Trace との連携**:
    Google Cloud の **Cloud Trace** と連携させ、リクエストごとのレイテンシ（遅延）の原因が「DBの検索時間」なのか「LLMの推論時間」なのかをミリ秒単位で特定する。

## 3. 本番環境での「ドリフト（精度劣化）」の監視

初期は精度が高くても、社内資料が更新されるにつれて精度が落ちることがある。

* **Vertex AI Model Monitoring**:
    回答の分布を監視し、特定のトピック（例: 新製品の問い合わせ）に対して「わかりません」という回答が急増していないか、あるいは回答のトーンが不安定になっていないかを検知する。
* **意味的ログ分析**:
    単なるエラーログではなく、ユーザーの質問をベクトル化してクラスター分析する。「どのジャンルの質問で失敗が多いか」を可視化し、不足している資料を特定する。

## 4. ユーザーフィードバック・ループの構築

3,000人の「目」を最大の評価リソースにする。

* **Implicit & Explicit Feedback**:
    * **Explicit**: 回答横の「👍/👎」ボタン。
    * **Implicit**: 回答後のリンククリック率、あるいは「その後、人間への問い合わせに繋がったか」のログ。
* **改善ルーチン**:
    「👎」がついたログを自動的に **Firestoreの「再学習・評価待ちコレクション」** へ飛ばす。エンジニアは毎週このリストを見て、チャンクの切り方やプロンプトを修正する。

## 5. パフォーマンスとコストの監視

3,000人が一斉に使うと、APIコストとクォータ（制限）が問題になる。

* **Cloud Monitoring ダッシュボード**:
    * **Token Usage**: 1ユーザーあたりの平均トークン消費量。
    * **Latency**: 99パーセンタイル（P99）のレスポンスタイム。
    * **Success Rate**: AIが「回答不能」と答えた割合。
* **監視のポイント**:
    特定のユーザーが異常に多いリクエストを投げていないか（Rate Limitの適用）、あるいは無駄に長いコンテキスト（[第4回](04_リランキング.md)）を渡しすぎてコストを圧迫していないかを監視し、動的にチャンク数を絞るロジックを検討する。

---

## 設計指針

RAGは「構築」よりも「運用中の微調整」が本番。**「どこが悪いか一目でわかる計器（ダッシュボード）」**を最初に作り、3,000人のフィードバックを糧に改善し続ける仕組み（LLMOps）を導入することが最終的な勝利条件。

## クイックスタート: ゴールデンデータセットの作成と評価実行

### 前提条件

- Genkit セットアップ済み（[第5回](05_Genkit.md)参照）

### 手順1: ゴールデンデータセット（JSONL形式）

```json
{"query": "ネジ999999の材質は？", "expectedAnswer": "SUS304", "category": "parts_catalog"}
{"query": "VPNに接続できない", "expectedAnswer": "パスワードの有効期限を確認してください", "category": "it_helpdesk"}
{"query": "有給休暇の申請方法は？", "expectedAnswer": "社内ポータルの勤怠管理画面から申請", "category": "hr_policy"}
```

!!! tip "データセットの規模"
    最低30件、理想は100件。カテゴリ（IT/部品/人事等）を均等に含めると偏りのない評価ができる。

### 手順2: Genkit Evaluator での評価実行

```typescript
import { genkit } from "genkit";
import { googleAI, gemini25Pro } from "@genkit-ai/googleai";

const ai = genkit({ plugins: [googleAI()] });

// 評価用データセットを読み込んで、RAG Flow の出力と比較する
// `npx genkit eval:run` コマンドで一括実行も可能

// カスタム評価基準の例: 引用番号が含まれているかチェック
function hasCitation(response: string): boolean {
  return /\[\d+\]/.test(response);
}
```
