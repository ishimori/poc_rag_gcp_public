第5回、いよいよ「RAGの心臓部」である**推論ロジック（Agentic Workflow）**の実装に入ります。

第4回までで「高品質な検索結果」を出す準備は整いました。しかし、ユーザーの質問が「PCが壊れた」のように絶望的に曖昧だった場合、どんなに優れた検索エンジンでも正解は出せません。ここで必要なのは、AIに**「検索する前に考え、足りなければ聞き返し、段階的に答えに近づく」**という自律的な振る舞いをさせることです。

Google Cloud の **Firebase Genkit** を使い、単なる一問一答を超えた「エージェント型RAG」を構築する設計要点を深掘りします。

## 第5回：Genkitによるエージェント・ワークフローの実装

従来のRAGは「入力 → 検索 → 生成」の直線（Linear）実行でしたが、第2弾で追求するのは**「ループと分岐（Agentic）」**を持つワークフローです。

### 1. インテント・ルーティング（意図による分岐）
3,000人規模の会社では、質問の性質が多岐にわたります。まずは「何についての質問か」をLLMに判定させ、最適な道具（ツール）を選ばせます。

* **実装の工夫**:
    Genkitの `defineTool` を使い、特定の検索エンジン（ITヘルプ用、ネジ在庫DB用、規定集用）を「道具」として定義します。
* **エンジニアの視点**:
    「ネジ番号 999999 について...」という入力があれば、AIは全文検索ではなく「部品マスターAPI」というツールを選択するようにプロンプトとスキーマを設計します。



### 2. クエリの具体化と「逆質問」のロジック
「質問が悪いから詳しく入力して」とユーザーに突き放すのではなく、AIが**「何が足りないか」を具体的に提示する**仕組みを実装します。

* **Self-Correction（自己修正）ループ**:
    1.  ユーザーの質問を解析し、検索に必要な「エンティティ（OS名、エラーコード、型番など）」が欠けていないか判定。
    2.  欠けている場合、検索をスキップして「OSはWindowsですか？Macですか？」とレスポンスを返す。
* **エンジニアの工夫**:
    Genkitの `Flow` 内で、`if` 分岐を使って「対話継続モード」と「検索実行モード」を切り替えます。これにより、無駄なベクトル検索コストを抑えつつ、ユーザー体験を向上させます。

### 3. マルチステップ・リーゾニング（多段階推論）
複雑な問い合わせ（例：「VPNが繋がらない。昨日までは使えていたが、設定変更はしていない」）に対して、AIに思考のステップを踏ませます。

* **ReActプロンプティング**:
    AIに「Thought（思考）」「Action（行動）」「Observation（観察）」のサイクルを回させます。
    * **Thought**: 「VPNの接続不可。まずは認証エラーかネットワークエラーかを確認する必要がある」
    * **Action**: 認証エラーに関するFAQを検索。
    * **Observation**: 検索結果から「パスワード期限切れ」の可能性が浮上。
    * **Final Answer**: 「パスワードの有効期限を確認してください」
* **Genkitでの実装**:
    `run` メソッドを使い、各ステップの出力をログとして残しながら、動的に次のステップを決定するパイプラインを構築します。



### 4. 状態管理（State Management）と履歴の保持
3,000人規模のヘルプデスクでは、一回の手続きが数往復に及ぶことがあります。

* **Firestoreによるセッション管理**:
    過去のやり取り（チャット履歴）をベクトル化して、今の質問とセットでLLMに渡します。
* **コンテキスト・圧縮**:
    履歴が長くなりすぎるとプロンプトの上限を超えるため、古い会話をLLMに要約（Summarize）させ、**「現在の状況のサマリー」**として保持し続けるエンジニアリングが、長期的な対話の精度を支えます。

### 5. Genkit Developer UI によるデバッグ
エージェント型ワークフローの最大の悩みは「AIがなぜその判断をしたか」がブラックボックス化することです。

* **Trace（トレース）の活用**:
    GenkitのUIを使えば、AIが「どのツールを選び」「どのプロンプトで」「どんな検索結果を受け取ったか」をタイムライン形式で可視化できます。
* **エンジニアの視点**:
    精度が出ない時は、このトレースを追い、「検索が悪い」のか「思考（ルーティング）が悪い」のかを特定し、そこだけをピンポイントで修正（リファクタリング）します。

## エンジニアとしての「設計の深み」

第1弾の失敗は、**「AIを単なる翻訳機（検索結果を整えるだけ）」**として使っていたことにあります。
第2弾では、**「AIを監督（検索の要否を判断し、結果を精査する）」**として設計します。
「質問が曖昧なら検索しない」という**勇気ある停止**をロジックに組み込むことが、結果としてユーザーからの信頼を勝ち取る（＝嘘を言わない）ことに繋がります。
