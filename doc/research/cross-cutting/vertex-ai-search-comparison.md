# 自前RAG vs Vertex AI Search: 比較分析

> 最終更新: 2026-03-22 | 関連DD: DD-022

## 背景

本PoCでは RAG パイプラインを全て自前実装している（チャンク分割、Embedding、ベクトル検索、リランキング等）。Google Cloud の Vertex AI Search（旧 Enterprise Search）を使えば、これらの多くがマネージドサービスとして提供される。

本ドキュメントでは、PoCで得た知見をもとに「自前実装で得られたもの」と「Vertex AI Search に移行した場合の得失」を整理する。次のPoCフェーズでのアーキテクチャ判断材料とする。

## 技術比較表

| 技術 | 自前実装 | Vertex AI Search | 備考 |
|------|:-------:|:----------------:|------|
| チャンク分割 | サイズ・overlap 自由に制御 | 自動（サイズ指定不可） | 「Bring Your Own Chunks」がプレビューで登場 |
| Embedding 生成 | モデル選択可 | 自動 | 自前 Embedding の持ち込みも可 |
| ハイブリッド検索 | 自前実装（RRF統合） | 組み込み済み | Vertex AI Search の方が実装コスト低 |
| リランキング | 自前実装 | 組み込み済み | 精度/リコールのバランス調整可 |
| メタデータブースト | 自前実装 | 対応 | freshness/metadata/popularity でブースト可能 |
| 権限フィルタ | `allowed_groups` で自前制御 | 対応 | IAM ベースのデータソースアクセス制御 |
| 検索チューニング | 評価データで手動調整 | 対応 | 訓練データ（corpus+query+labels）をアップロード |
| 回答カスタマイズ | プロンプト自前設計 | 対応 | スタイル・トーン・長さを制御可 |
| Contextual Retrieval | LLM で文脈説明を自動付与 | **不可** | BYOCで文脈付与済みチャンクを投入すれば限定的に可能だが、パイプラインの複雑性は自前と同等になる |
| 曖昧質問の聞き返し | LLM で曖昧判定 + 確認質問生成 | **検索外で実装可能** | 検索エンジンの機能ではなくオーケストレーション層の機能。VAISをRetrieverとして使い外側で実装可能 |
| Shadow Retrieval | 権限除外を検出して明示的に拒否 | **検索外で実装可能** | 同上。VAISの検索結果に対して自前ロジックで判定可能 |
| Answerability Gate | スコア閾値で回答を拒否 | **検索外で実装可能** | 同上。検索後の制御層として実装可能 |
| チャンクサイズ実験 | 複数コレクションで A/B 比較 | **限定的** | BYOCを利用すれば限定的に可能だが、自前ほどの自由度はない |

## 本質的な比較軸

技術比較表は「機能の有無」を整理したものだが、アーキテクチャ判断においてはより本質的な軸が存在する。

### 制御可能性（Controllability）

| 項目 | 自前 | Vertex AI Search |
|------|:----:|:----:|
| Retrieval 内部の透明性 | 高 | 中（低レベル制御不可） |
| 実験自由度 | 高 | 低 |
| 原因分析 | 可能 | 限定的 |
| パラメータ調整粒度 | 完全 | ブースト・チューニングのみ |

> **注**: Vertex AI Search は「完全ブラックボックス」ではない。チューニングデータ（query + relevance）のアップロード、Boost 設定（freshness / metadata）、BYOC（Bring Your Own Chunks, プレビュー）により一定の制御は可能。ただし、RRF の重み調整やスコア計算式の変更といった低レベル制御はできない。

### 再現性（Determinism）

| 項目 | 自前 | Vertex AI Search |
|------|------|------|
| スコア計算式 | 明確・固定可能 | 内部モデル更新の影響を受ける可能性 |
| 閾値制御 | 完全 | 限定的 |
| ロジック固定 | 可能 | Google 側の更新で変動しうる |

エンタープライズ用途では、同じクエリに対して同じ結果が得られる再現性が重要となる。

### コスト構造

| 項目 | 自前 | Vertex AI Search |
|------|------|------|
| 初期構築 | 高（開発工数） | 低（数クリック） |
| 検索コスト | Firestore 読み取り + Cloud Run | クエリ単価（検索ユニット課金） |
| インデックス維持 | Firestore ストレージ | インデックス維持費 |
| スケール時 | 設計次第で可変 | 比較的予測可能 |

小規模文書（現状 61 ファイル）では自前が安価だが、大量の小規模ドキュメントを扱う場合は VAIS の「検索ユニット」課金が重くなるケースもある。規模に応じた試算が必要。

## スコア推移から見る分析

本PoCでの精度改善の推移:

| 段階 | Overall | 追加した技術 | Vertex AI Search で代替可能か |
|------|---------|------------|:---:|
| 素のベクトル検索 | ~50% | ベクトル検索のみ | 可能（同等レベル） |
| +リランキング+ハイブリッド | ~70% | リランキング、キーワード検索併用 | 可能（組み込み済み） |
| +contextual_retrieval+メタデータ | ~80% | LLM文脈付与、メタデータスコアリング | **一部不可**（文脈付与は自前のみ） |
| +権限制御+聞き返し+チャンク最適化 | 85%+ | Shadow Retrieval、曖昧判定、チャンクサイズ調整 | **検索エンジン単体では不可**（オーケストレーション層の領域） |

### 考察

- **~70% まで**: Vertex AI Search でも到達可能。セットアップも大幅に楽（数クリック）
- **70%→80%**: Contextual Retrieval は自前のみ。ただし Vertex AI Search の検索チューニング機能（訓練データアップロード）で代替できる可能性あり
- **80%→85%+**: 聞き返し、Shadow Retrieval、Answerability Gate は検索エンジンの機能ではなくオーケストレーション層の設計。Vertex AI Search を Retriever として使い、外側に自前制御層を組むハイブリッド構成でも実現可能

### 重要な知見: 精度は検索エンジンではなく制御層が決める

PoCを通じて得られた最も重要な知見は、**80%→85%+ の精度改善は検索エンジンの性能差ではなく、検索をどう制御するか（オーケストレーション設計）によって決まる**ということである。

聞き返し・Shadow Retrieval・Answerability Gate はいずれも「検索結果をどう解釈し、どう処理するか」のロジックであり、検索エンジンの内部処理とは独立している。この理解により、「自前 vs マネージド」の二択ではなく、**検索基盤と制御層を分離して最適化する**という設計判断が可能になる。

## Ingest パイプラインの比較

### 自前（現状）

```
ソース MD → チャンク分割 → 文脈説明付与(LLM) → Embedding 生成 → Firestore 保存
              ローカル処理     Gemini API×N回      Vertex AI API     Firestore API
              （一瞬）         （時間の90%）        （バッチで速い）   （速い）
```

- 61ファイル / chunk_size=1200 で約25分（contextual_retrieval ON）
- 時間の支配要因は「文脈説明付与」のLLM呼び出し（チャンクごとに1回）
- **留意**: Vertex AI Search に移行しても、Contextual Retrieval（事前処理）を継続するなら、この「25分」は短縮されない。運用工数は減るが、パイプラインの実行時間は自前レイヤーが残る限り変わらない

### Vertex AI Search

```
ソース文書 → アップロード → （自動で全て処理）
              GCS に配置     チャンク分割 + Embedding + インデックス構築
```

- ドキュメントを GCS にアップロードするだけ
- 分〜時間単位で自動インデックス構築
- チャンクサイズやEmbeddingモデルの選択は不可（自動決定）

## 自前実装で得られた知見（Vertex AI Search では得られないもの）

1. **精度改善の因果関係の理解**: DD-019-8 で「ベクトル検索段階で正しい文書が引けない」と特定し、DD-019-9 でチャンクサイズ調整が有効と実証。Vertex AI Search では低レベルの原因特定が困難

2. **技術の効果測定**: アブレーションスタディ（DD-013-2）で各技術の寄与を定量評価。「リランキングは+15%」「メタデータスコアリングは+2%」等の知見は、マネージドサービスでは得られない

3. **ドメイン特化の最適化**: 社内文書の特性（権限レベル、カテゴリ構造、曖昧な質問パターン）に合わせた細かい調整ができた

4. **障害の根本原因分析**: DD-019-14 で「キーワード検索に権限フィルタがない」というバグを発見・修正。マネージドサービスでは検索結果が「意図せず漏洩している」のか「スコアリングの結果」なのかの判別が困難であり、セキュリティ要件が厳しいシステムではこの「デバッグ可能性」が自前実装の強い優位点となる

## 次のPoCへの推奨事項

### Vertex AI Search を採用すべきケース

- ドキュメント量が大規模（数万件〜）でスケーラビリティが必要
- チューニングに時間をかけられない（「それなりの精度」で十分）
- 多言語対応が必要
- インフラ管理を最小化したい

### 自前実装を継続すべきケース

- 80%以上の精度が求められる
- ドメイン特化の機能が必要（聞き返し、権限制御の細かい制御等）
- 精度改善の原因分析が必要（なぜ精度が上がったか/下がったかを理解したい）
- チャンクサイズ等のパラメータ実験が必要

### ハイブリッドアプローチ（推奨）

Vertex AI Search をベース検索エンジンとして使いつつ、自前レイヤーで付加価値を追加する:

```
ユーザー質問
  → 曖昧判定（自前LLM）
  → Vertex AI Search で検索
  → Shadow Retrieval チェック（自前）
  → Answerability Gate（自前）
  → LLM 回答生成（自前プロンプト）
```

これにより:
- 検索基盤の運用コストを削減（Vertex AI Search に委譲）
- 差別化ポイント（聞き返し、権限制御、品質ゲート）は自前で維持
- 80%→85%+ の精度改善ロジックを保持
