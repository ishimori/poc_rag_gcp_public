# メタデータ設計

> 各チャンクに付与するメタデータの一覧と用途。インジェスト（第1回）から検索（第3回）、権限管理（第6回）まで横断的に使用される。

---

## メタデータフィールド一覧

| フィールド | 型 | 用途 | 設定タイミング | 参照先 |
|-----------|-----|------|--------------|--------|
| `doc_id` | string | 元ファイルへの紐付け。チャンク更新時のグルーピングに使用 | インジェスト時 | [第1回](../01_データ前処理.md) |
| `parent_doc_id` | string | 同一PDFから生成された全チャンクの親ID。アトミック更新に使用 | インジェスト時 | [第1回](../01_データ前処理.md) |
| `content_hash` | string | ファイル内容のSHA-256。冪等性担保（重複防止） | インジェスト時 | [第1回](../01_データ前処理.md) |
| `source_url` | string | 元ファイルへのGCSパス。引用表示に使用 | インジェスト時 | [第1回](../01_データ前処理.md) |
| `page_number` | number | 引用元ページ番号 | インジェスト時 | [第1回](../01_データ前処理.md) |
| `doc_version` | string | データの鮮度管理用バージョン | インジェスト時 | [第1回](../01_データ前処理.md) |
| `category` | string | `it_helpdesk`, `parts_catalog`, `hr_policy` 等。検索範囲の限定に使用 | インジェスト時 | [第2回](../02_チャンキング戦略.md) |
| `security_level` | string | `general`, `admin_only` 等。権限フィルタリングに使用 | インジェスト時 | [第2回](../02_チャンキング戦略.md) |
| `allowed_groups` | string[] | 閲覧可能グループIDの配列。Pre-filteringに使用 | インジェスト時 | [第6回](../06_セキュリティ.md) |
| `updated_at` | timestamp | 最新情報の優先度付け | インジェスト時 | [第2回](../02_チャンキング戦略.md) |
| `chunk_index` | number | 同一ドキュメント内のチャンク順序 | チャンキング時 | [第2回](../02_チャンキング戦略.md) |
| `doc_title` | string | ヘッダー・インジェクションで注入する親ドキュメントタイトル | チャンキング時 | [第2回](../02_チャンキング戦略.md) |

## 設計ガイドライン

1. **インジェスト時に全て付与する** — 後から付け足すのは運用コストが高い
2. **`category` は検索速度に直結** — Firestore の `where` 句で物理的に探索範囲を絞るため、適切な粒度で設計する
3. **`allowed_groups` はPre-filteringの生命線** — Post-filtering（検索後に権限チェック）はアンチパターン（[第6回](../06_セキュリティ.md)参照）
4. **`content_hash` で重複を防ぐ** — 同じファイルの再アップロードでチャンクが重複しないようにする
