## 第1回：技術深掘り編 — 構造化インジェストとデータ・クレンジング

RAGの精度を決定付けるのは、モデルに渡す「情報の密度」です。PDFの非構造データやExcelの半構造データを、いかに**「意味的整合性を保ったまま」**ベクトル化するかが鍵となります。

### 1. Document AI による「セマンティック・パース」
単なるOCR（文字認識）では、段落の順序が入れ替わったり、表データがただの文字列の羅列になったりします。

* **Layout Parser / Form Parser の利用**:
    * Google Cloudの **Document AI** を使用し、ドキュメントを「Visual Elements」として解釈します。
    * **エンジニアの工夫**: 抽出結果のJSONから、`h1`, `h2` などのヘッダー構造を維持したまま **Markdown形式** に変換します。LLMはMarkdownの構造（`#`, `*`, `|---|`）を極めて正確に理解するため、検索ヒット時の回答精度が劇的に向上します。
* **マルチモーダル処理 (Gemini 1.5 Flash)**:
    * 図解やフローチャートが含まれる場合、テキスト化だけでは情報が欠落します。
    * **エンジニアの工夫**: 画像部分を切り出し、Gemini 1.5 Flash等に「この図は何を説明しているか」をプロンプトで記述（キャプション生成）させ、そのテキストを元のコンテキストに埋め込みます。

### 2. データ・デンマライゼーション（非正規化）戦略
DBやExcelのデータをそのままベクトル化するのはアンチパターンです。

* **コンテキストの注入**:
    * `part_id: 123, material: steel` というローデータに対し、インジェスト時にメタデータを結合（Join）し、「文章」として展開します。
* **メタデータのエンリッチメント**:
    * Firestoreへ格納する際、単なるベクトル（Embedding）だけでなく、以下のメタデータを付与します。
        * `source_url`: 元ファイルへのポインタ
        * `page_number`: 引用元ページ
        * `category`: IT / ネジ / 規定 などのタグ（第2回で解説する「フィルタリング」に必須）
        * `doc_version`: データの鮮度管理用

### 3. テクニカル・スタック：イベント駆動型パイプライン
3000人規模の更新頻度に対応するため、スケーラブルな構成を組みます。

1.  **GCS (Cloud Storage)**: `gs://raw-bucket/` にファイルが着地。
2.  **Eventarc / GCS Trigger**: ファイル作成イベントを検知し、**Cloud Run** を起動。
3.  **Cloud Run (Processing)**: 
    * Document AI API をコールして構造化。
    * Pythonの `LangChain` や `LlamaIndex` 等のライブラリを用いて、論理的な区切りでチャンク分割（詳細は第2回）。
4.  **Vertex AI Embedding API**: 
    * `text-multilingual-embedding-002`（2026年時点の推奨）を使用。
    * 日本語の技術用語や型番に対しても高い次元圧縮精度を持つモデルを選択。
5.  **Firestore (Vector Search)**: 
    * ベクトルとメタデータを一括保存。

### 4. 冪等性（Idempotency）と一貫性の担保
* **Content-Addressable Storage**: 
    * ファイルの内容から計算した `hash(SHA-256)` をドキュメントIDとして使用します。
    * これにより、同じ内容のファイルが再アップロードされても、ベクトルDB内に重複データが作られず、検索ノイズを最小限に抑えられます。
* **アトミックな更新**:
    * 1つのPDFから100個のチャンクが生成される場合、一部のチャンクだけが古いまま残るのを防ぐため、`parent_doc_id` でグルーピングし、更新時は全チャンクを入れ替えるロジックを実装します。

## エンジニアへのアドバイス：第1回の落とし穴

「精度が出ない」というチームの多くは、**「PDFをただの長い文字列として、1000文字ごとに機械的に切っている」**ことが多いです。これでは、文章の途中で意味が分断され、ベクトル検索が機能しません。

第1回での最重要課題は、**「情報の意味的な最小単位（チャンク）を、メタデータ付きの綺麗なMarkdownとして切り出すパイプラインを自動化すること」**です。
