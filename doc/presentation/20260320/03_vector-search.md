## 従来のキーワード検索との違い

従来の検索は「同じ言葉が含まれているか」で探します。
「PCが重い」と検索すると、「PCが重い」という文字列を含む文書しか見つかりません。

しかし実際には、「パソコンの動作が遅い」「起動に時間がかかる」「メモリ不足」なども関連する情報です。
ベクトル検索（Vector Search）は、言葉が違っても**意味が近ければ見つけ出す**ことができます。

## どうやって「意味」を比べるのか

ベクトル検索は、以下の2つのステップで動きます。

### ステップ1：文章を数値の列に変換する

AI（Embedding（埋め込み）モデル）が、文章を**768個の数値の列**に変換します。
この数値の列を「ベクトル」と呼びます。

意味が似た文章は、似た数値の列になります。

- 「PCが重い」→ [0.12, -0.45, 0.78, ... ] （768個の数値）
- 「パソコンの動作が遅い」→ [0.11, -0.43, 0.76, ... ] （近い数値になる）
- 「今日の天気」→ [0.89, 0.23, -0.56, ... ] （全く違う数値になる）

### ステップ2：数値の「距離」で似ている度合いを測る

質問文もベクトルに変換し、保存済みの全チャンクのベクトルと「距離」を比較します。
距離が近いほど、意味が似ていると判断されます。

```mermaid
flowchart LR
    A["ユーザーの質問\n「PCが重い」"] --> B["Embedding API\n（768次元ベクトルに変換）"]
    B --> C["Firestore\nベクトル検索"]
    C --> D["距離が近い順に\nチャンクを取得"]
    D --> E["上位5件を\n次のステップへ"]
```

## 本PoCでの実装

- **Embeddingモデル**: Google Cloud の gemini-embedding-001 を使用
- **ベクトルの次元数**: 768次元（768個の数値で意味を表現）
- **データベース**: Firestore（ファイアストア）のベクトル検索機能を使用
- **検索件数**: 上位5件を取得し、次のリランキング（第4回）に渡す

## ベクトル検索の強みと弱み

### 強み：言い換えに強い

| ユーザーの質問 | 見つけられる文書 |
|-------------|--------------|
| PCが重い | 「パソコンの動作が遅い場合の対処法」 |
| VPNが繋がらない | 「リモート接続のトラブルシューティング」 |
| 有給の取り方 | 「年次有給休暇の申請手順」 |

### 弱み：型番・品番の区別が苦手

ベクトル検索は「意味」で探すため、**数字の1桁違い**を正確に区別できません。

- 「ネジ999999」と「ネジ999998」は、意味的にはほぼ同じ「ネジの番号」
- しかし、業務上はまったく別の部品

この弱点を補うのが、第6回で紹介する「ハイブリッド検索」です。

## まとめ

ベクトル検索は、言葉が違っても意味が近い情報を見つけ出せる強力な技術です。
「意味の近さ」を768個の数値で表現し、数学的に距離を測ることで実現しています。
一方で、型番のような厳密な一致が必要な検索には弱点があり、他の技術との組み合わせが重要です。
