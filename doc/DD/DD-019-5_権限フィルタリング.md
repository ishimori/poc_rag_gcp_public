## 目的

ベクトル検索にPre-filteringを導入し、ユーザーの権限に応じたドキュメントのみを検索結果に返すようにする。securityテストタイプ（現在5件スキップ、0%）を改善する。

## 背景・課題

現在のRAGパイプラインは全ドキュメントを無差別に検索しており、一般社員が役員会議事録（`allowed_groups: ["exec_board"]`）や給与規定（`security_level: admin_only`）の内容を取得できてしまう。

### 設計方針: Pre-filtering

調査資料（`doc/research/06_セキュリティ.md`）に基づき、**Pre-filtering**（検索クエリ発行時点で権限フィルタ適用）を採用する。

```
Post-filtering（❌）: 全件検索 → 上位10件 → 権限チェック → 権限外を除外 → コンテキストがゼロになるリスク
Pre-filtering（✅）: 権限フィルタ付き検索 → 権限内の上位10件のみ返却
```

Firestore Vector Search の `where` 句で `allowed_groups` に `array-contains-any` を適用する。

## 検討内容

### PoCでのユーザーグループ指定

本番では Firebase Auth Custom Claims からユーザーのグループを取得するが、PoCでは認証基盤が未構築。以下の方式で代替する:

| 方式 | メリット | デメリット |
|------|---------|-----------|
| A: configで固定指定 | シンプル | デモで切替できない |
| **B: UIセレクトボックス + APIパラメータ** | デモで直感的に切替可能 | UI側の改修が必要（軽微） |
| C: Firebase Auth統合 | 本番同等 | PoCスコープ外 |

**→ 方式B を採用。** チャット画面のサイドバーに「ロール選択」セレクトボックスを追加し、選択したロールの `user_groups` をAPIリクエストに含める。configにはデフォルト値（`["all"]`）を持たせ、APIパラメータがあればそちらを優先する。

#### ロール定義（PoC用）

| ロール名 | user_groups | 説明 |
|---------|-------------|------|
| 一般社員 | `["all"]` | 公開文書のみ閲覧可能 |
| HR管理者 | `["all", "hr_admin"]` | 公開文書 + 給与規定 |
| 役員 | `["all", "exec_board"]` | 公開文書 + 役員会議事録 |

**重要: 全ロールに `"all"` を含める。** `allowed_groups: ["all"]` の公開文書に `array_contains_any` でマッチさせるため、`"all"` がないと公開文書が閲覧できなくなる。

### `salary_policy.md` の `allowed_groups` 未設定

現在 `salary_policy.md` には `security_level: admin_only` はあるが `allowed_groups` が未設定（デフォルト `["all"]` になる）。`allowed_groups: ["hr_admin"]` を追加する必要がある。

### SearchResult への `allowed_groups` 追加

現在の `SearchResult` は `security_level` のみ保持。フィルタリングはFirestoreクエリ側で行うため、`SearchResult` への `allowed_groups` 追加は不要（検索結果に返る時点で権限チェック済み）。

### rag_flow への統合箇所

```
Step 0: 曖昧判定（Pre-RAG）
Step 1: ベクトル検索 ← ここでPre-filtering適用
Step 2: リランキング
Step 2.5: メタデータスコアリング
Step 3: コンテキスト構築
Step 4: LLM回答生成
```

Pre-filteringはStep 1の`vector_search()`内部で適用するため、`flow.py`の変更は最小限（`user_groups`の受け渡しのみ）。

### 権限外ドキュメントへのアクセス時の回答

Pre-filteringにより権限外ドキュメントは検索結果に含まれない。LLMには権限外の情報が渡されないため、既存のシステムプロンプト（「コンテキストに情報がない場合は『提供された情報には記載がありません』と回答」）で適切に処理される。

ただし、セキュリティテストの期待回答は「アクセス権限がありません」等の権限拒否メッセージ。これを実現するには:
- **案1**: 検索結果ゼロ件かつ権限フィルタONの場合、LLMに「権限制限により情報が提供できない可能性がある」旨をヒントとして渡す
- **案2**: システムプロンプトに「検索結果がない場合は権限不足の可能性に言及する」を追加

**→ 案1を採用。** `flow.py` で検索結果が0件かつ `permission_filter` がONの場合、コンテキストに権限制限の注記を追加する。

### 検索結果0件の原因区別

検索結果が0件になる原因は2つある:
- **A**: 権限フィルタで制限付きドキュメントが除外された（→ 権限拒否メッセージが適切）
- **B**: そもそも関連ドキュメントが存在しない（→ 「記載がありません」が適切）

厳密に区別するには権限フィルタなしで再検索が必要（コスト2倍）。PoCではシステムプロンプトに**「検索結果がない場合、アクセス権限による制限の可能性もあります」**とヒントを渡し、判断をLLMに委ねる。これにより:
- 給与テーブルの質問 → LLMが「権限」に言及する回答を生成（期待通り）
- 月の土地の質問 → LLMが「情報がありません」と回答（文脈的に権限とは無関係と判断）

完全な区別は本番化時に検討（権限フィルタなしのカウントクエリ等）。

### 権限フィルタ済みだが関連文書が残るケース

security-001「給与テーブルを見せて」で `salary_policy.md` がフィルタ除外されても、他のHR文書（`leave_policy.md` 等）がヒットする可能性がある。その場合LLMは有休規定を元に回答し、権限拒否メッセージにはならない。

**これはPre-filteringの設計上正しい動作。** ユーザーの権限内の情報で回答するのがPre-filteringの本質であり、「この情報は権限外です」と明示するのはPost-filteringの責務。PoCではこの動作を許容し、テスト期待値側で評価する。
