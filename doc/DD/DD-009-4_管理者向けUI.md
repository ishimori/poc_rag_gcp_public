# DD-009-4: 管理者向けUI（ドキュメント管理・チューニングダッシュボード）

| 作成日 | 更新日 | ステータス |
|--------|--------|------------|
| 2026-03-20 | 2026-03-20 | 進行中 |

## 目的

文書のアップロード、インジェスト結果の確認、評価の実行・結果表示、パラメータチューニングを行える管理者向けUIを構築する。現在のCLI操作（`scripts/ingest.py`, `scripts/evaluate.py`）を画面化し、非エンジニアの管理者でも運用できるようにする。

## 背景・課題

- 現在の文書取り込みは `uv run python scripts/ingest.py` のCLI実行が必要
- 評価も `uv run python scripts/evaluate.py` で手動実行
- 本番構成で「GCSにファイルを置くだけ」は管理として雑
- 管理者が「文書を投入したら結果を確認し、チューニングし直す」サイクルを回せるUIが必要
- DD-009-3でTypeScript移行を決定しており、同一プロジェクトで管理画面も構築できる

## Phase 0 精査結果

### PoCスコープの判断

PoCフェーズでは「パラメータを変えて→再取り込み→再評価→スコア比較」のサイクルを回せることがコアバリュー。フル機能の管理画面は不要。

#### 採用する機能（MVP）

| # | 機能 | 現状（CLI） | 管理画面で |
|---|------|-----------|-----------|
| 1 | **インジェスト実行** | `scripts/ingest.py` を手動実行 | ボタンクリックで実行 |
| 2 | **評価実行** | `scripts/evaluate.py` を手動実行 | ボタンクリックで実行 |
| 3 | **評価結果表示** | `results/` のJSONを目視 | タイプ別スコア表、失敗ケース詳細 |
| 4 | **パラメータ調整** | `src/config.py` を手動編集 | 画面上でチャンクサイズ・top_k・閾値等を変更 |
| 5 | **再チューニング** | パラメータ変更→ingest→evaluate を手動 | パラメータ変更→ワンクリックで再インジェスト→再評価→スコア比較 |

#### PoCでは削るもの

| 機能 | 削る理由 |
|------|---------|
| 文書アップロードUI | `test-data/sources/` は固定テストデータ。PoCで動的アップロードは不要 |
| 文書一覧画面 | ~~DD-009-2（データブラウザ）と重複~~ → データブラウザをPhase 3として本DDに統合 |
| 管理者認証 | PoCなので不要 |

### 画面構成（MVP確定）

```
管理画面（React Router で既存チャットUIと統合）
├── /admin — ダッシュボード
│   ├── 最新の評価スコア（全体 + タイプ別）
│   ├── 登録文書数・チャンク数
│   └── 現在のパラメータ値
├── /admin/tuning — 評価 & チューニング
│   ├── パラメータ設定（chunk_size, chunk_overlap, top_k, rerank_top_n, rerank_threshold）
│   ├── インジェスト実行ボタン
│   ├── 評価実行ボタン
│   ├── 再チューニング（ingest→evaluate ワンクリック）
│   └── 最新の評価結果（タイプ別スコア表、失敗ケース詳細）
├── /admin/data — データブラウザ（DD-009-2 から統合）
│   ├── Firestoreチャンク一覧（source_file, chunk_index, category, security_level, content先頭50文字）
│   ├── 行クリックでcontent全文を詳細表示
│   ├── カテゴリ・セキュリティレベルでフィルタ
│   └── 「データ取得」ボタン（API経由でFirestoreから取得、embeddingカラムは除外）
└── /admin/history — 評価履歴
    ├── 過去の評価結果一覧（日時順）
    └── スコア推移の比較
```

### アーキテクチャ

- DD-009-3のTypeScriptフロントエンドと同一プロジェクト（React Router導入）
- 利用者向け（`/` チャット）と管理者向け（`/admin/*`）はルーティングで分離
- バックエンドAPI: `admin` Cloud Function（パスベースルーティング、timeout 540秒）
  - `POST /api/admin/ingest` — インジェスト実行（`clear` オプション付き）
  - `POST /api/admin/evaluate` — 評価実行（レポート自動保存）
  - `GET /api/admin/evaluate/results` — 評価結果一覧取得（新しい順）
  - `GET /api/admin/config` + `PUT /api/admin/config` — パラメータ読み書き
  - `GET /api/admin/chunks` — チャンク一覧取得（embedding除外、フィルタ対応）
- ローカル開発: chat(8081) + admin(8082) + Vite(5180) の3プロセス構成

### 長時間処理のタイムアウト設計

ingest・evaluate は数分かかる可能性がある。PoCではシンプルさ優先で **Cloud Functions Gen2 のタイムアウト延長**（最大60分）で対応する。非同期ジョブキューやCloud Run Jobsへの切り出しは本番化フェーズで検討。

## Phase 1 機械検証手順

開発サーバー起動後（`bash scripts/dev.sh`）、以下のcurlで各APIを確認する:

```bash
# 1. パラメータ取得
curl -s http://localhost:8082/config | jq .

# 2. パラメータ更新
curl -s -X PUT http://localhost:8082/config \
  -H "Content-Type: application/json" \
  -d '{"chunk_size": 600, "top_k": 8}' | jq .

# 3. パラメータが反映されたか確認
curl -s http://localhost:8082/config | jq .

# 4. インジェスト実行（clearオプション付き）
curl -s -X POST http://localhost:8082/ingest \
  -H "Content-Type: application/json" \
  -d '{"clear": true}' | jq .

# 5. チャンク一覧取得
curl -s "http://localhost:8082/chunks?limit=3" | jq .

# 6. チャンク一覧（フィルタ付き）
curl -s "http://localhost:8082/chunks?category=it_helpdesk&limit=3" | jq .

# 7. 評価実行（数分かかる）
curl -s -X POST http://localhost:8082/evaluate | jq .overall

# 8. 評価結果一覧
curl -s http://localhost:8082/evaluate/results | jq '.[0].overall'

# 9. パラメータを元に戻す
curl -s -X PUT http://localhost:8082/config \
  -H "Content-Type: application/json" \
  -d '{"chunk_size": 800, "top_k": 10}' | jq .
```
