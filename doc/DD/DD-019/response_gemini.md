RAGシステムのPoCにおいて、Overall 81.1%というスコアは非常に素晴らしい進捗です。ベースラインの28.4%からここまで引き上げたのは、ハイブリッド検索、Embeddingの`task_type`指定、メタデータスコアリングなど、各手法を論理的かつ着実に積み重ねた結果ですね。特にDD-019-8での25.7ptアップは、Vertex AIやGeminiの仕様を的確に突いた見事な改善だと思います。

ここから90%の壁を越え、残るエッジケースを潰していくための具体的な改善策を提案します。

### 1. security 0% 改善策（権限不足の明示）

LLMが「検索結果がない」のか「権限で弾かれた」のか区別できないという、Pre-filtering特有のジレンマですね。これを解決するには、検索フェーズで「権限の有無」を検知する仕組みを挟む必要があります。

- **対策名**: 2段階検索（権限チェックフェーズの分離）
- **概要**: ユーザーのクエリに対して、まず「フィルタなし（権限無視）」で軽量なベクトル検索を行い、ヒットした上位文書のメタデータ（`allowed_groups`）を取得します。ユーザーがその権限を持っていなければ、ドキュメントの本文は取得せず、LLMに「権限が必要な文書が存在しますが、アクセス権がありません」というメタ情報のみを渡して拒否回答を生成させます。権限があれば通常のPre-filtering検索に進みます。
- **期待効果**: security 100% (5/5) への完全な改善
- **実装難易度**: 中
- **再Ingestの要否**: 不要
- **トレードオフ・リスク**: ベクトル検索（またはFirestoreへのクエリ）が追加で1回走るため、レイテンシがわずかに増加します。

### 2. semantic 67% 改善策（意味的検索の強化）

すでにハイブリッド検索や`task_type`指定を行っているにも関わらず引けない4件は、ユーザーのクエリとドキュメントの語彙（ボキャブラリ）の間に、Embeddingだけでは埋めきれないギャップがある可能性が高いです。



- **対策名**: HyDE (Hypothetical Document Embeddings) の導入
- **概要**: クエリから直接ベクトル検索するのではなく、Pre-RAGフェーズで一度LLMに「クエリに対する仮の回答（Hypothetical Document）」を生成させます。その「仮の回答」のベクトルを生成し、ドキュメントのベクトルと照合させます。
- **期待効果**: semanticカテゴリの残り4件の改善。ユーザーの短い曖昧な語彙が、マニュアルに特有の専門用語や詳細な表現に拡張されるため、ヒット率が向上します。
- **実装難易度**: 中
- **再Ingestの要否**: 不要
- **トレードオフ・リスク**: 検索前にLLMの生成処理（Flash等の高速モデルを使用推奨）が挟まるため、全体の応答速度が低下します。

### 3. ambiguous 60% 改善策（LLM判定の安定化）

LLM-as-JudgeやPre-RAGでの曖昧判定がブレるのは、LLMにとって「何をもって曖昧とするか」の基準が抽象的すぎるためです。

- **対策名**: Few-ShotプロンプティングとCoT (Chain of Thought) の導入
- **概要**: Clarifier（曖昧判定ロジック）とLLM-as-Judgeのプロンプトに、「明確な質問」と「曖昧な質問」の具体的なペアを3〜5パターン提示します（Few-Shot）。さらに、判定を出力する前に「なぜ曖昧なのか/情報が足りないのか」の理由を思考プロセスとして言語化（CoT）させてから、最終的なフラグを出力させます。
- **期待効果**: ambiguous判定の再現性向上、ブレの解消（100%に近づく）
- **実装難易度**: 低
- **再Ingestの要否**: 不要
- **トレードオフ・リスク**: プロンプト長と思考プロセスの出力分、トークン数と評価コストが微増します。

### 4. unanswerable 80% 改善策（過剰回答の防止）

LLMが「何とかして答えてあげよう」と親切心を発揮してしまうハルシネーションです。システムプロンプトでの指示だけでは限界があります。

- **対策名**: 事後検証（Self-Correction / Critique）パイプラインの追加
- **概要**: 生成した回答をそのままユーザーに返すのではなく、出力前に「この回答は、検索されたコンテキスト[A, B, C]の内容のみから導出されているか？」をチェックするステップをプロンプト内（または別の軽量LLMコール）に追加します。厳密な推論チェック（NLI）を行い、外部知識が混ざっている場合は「該当情報なし」にフォールバックさせます。
- **期待効果**: unanswerable 100% への改善、および全体的なハルシネーションの撲滅
- **実装難易度**: 中
- **再Ingestの要否**: 不要
- **トレードオフ・リスク**: 出力トークン数の増加と、それに伴うレイテンシ・コストの増加。

### 5. 90%以上に引き上げるための見落としている技術

現状の「800文字/150文字重複」という機械的な分割は、ベースラインとしては優秀ですが、意味の途中で分断されるリスクがあります。

- **対策名**: セマンティック・チャンキング（Semantic Chunking）
- **概要**: 文字数ではなく、Markdownの構造（見出し、段落、リスト単位）や、意味のまとまりごとにチャンクを分割します。LlamaIndexなどのフレームワークにあるNodeParserを利用するか、正規表現でセクションごとに分割します。
- **期待効果**: semantic, multi_chunk, cross_category など、文脈理解が必要なカテゴリ全体の底上げ（数ポイントの改善）
- **実装難易度**: 中
- **再Ingestの要否**: 要
- **トレードオフ・リスク**: 対象の61ファイル（.md）のフォーマットが統一されていない場合、パース処理の調整に手間がかかります。

### 6. アーキテクチャ全体の構造的改善点

現状のパイプラインは「Pre-RAG → 検索（ハイブリッド） → 後処理 → 生成」と直列で、非常に重厚になっています。PoCとしては素晴らしいですが、本番稼働を見据えると柔軟性と保守性に課題が出る可能性があります。



- **対策名**: Agentic RAG（ツール呼び出し）への移行
- **概要**: 直列のパイプラインを廃止し、LangGraphなどのフレームワークやGeminiのFunction Calling（Tool Use）を活用したエージェント型に変更します。「ベクトル検索ツール」「キーワード検索ツール」「権限確認ツール」を用意し、ユーザークエリの意図（Intent）に応じてLLM自身に「どのツールを、どの順番で使うべきか」を判断させます。
- **期待効果**: クエリに不要な処理（例：明確な型番検索にベクトル検索や曖昧判定を走らせる等）をスキップできるため、トータルの精度とレイテンシの最適化が可能になります。
- **実装難易度**: 高
- **再Ingestの要否**: 不要
- **トレードオフ・リスク**: アーキテクチャのパラダイムシフトとなるため、実装とデバッグの難易度が大きく跳ね上がります。まずは直列で90%を目指してから着手する方が安全です。
