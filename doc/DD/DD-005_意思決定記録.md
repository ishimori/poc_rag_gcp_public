## 目的

リサーチドキュメントで提示された技術選択肢について、このプロジェクトとしての採用/不採用を「選択肢 → 採用 → 理由」の形式で記録し、後から参加したメンバーが意思決定の経緯を理解できるようにする。

## 背景・課題

- 現状のドキュメントは「AとBという選択肢がある」という紹介に留まっており、「このプロジェクトではどちらを選んだのか」が不明
- 技術選定の理由が暗黙知のまま残ると、後からの変更判断ができない
- ADR（Architecture Decision Record）として残すことで、将来の見直し時に「なぜこうなっているか」を参照できる

### ADRのフォーマット

```markdown
## ADR-001: [テーマ]

- **ステータス**: 採用 / 検討中 / 却下
- **日付**: YYYY-MM-DD
- **選択肢**:
  1. A — メリット / デメリット
  2. B — メリット / デメリット
- **決定**: Aを採用
- **理由**: ...
- **影響**: この決定により〜が必要になる
```

### やらないこと
- まだ決まっていない項目の無理な決定（「検討中」ステータスで残す）
- リサーチドキュメント本文の変更（ADRは独立ドキュメントとして作成）
