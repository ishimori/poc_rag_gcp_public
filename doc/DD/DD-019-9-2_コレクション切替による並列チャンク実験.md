# DD-019-9-2: コレクション切替による並列チャンク実験

| 作成日 | 更新日 | ステータス |
|--------|--------|------------|
| 2026-03-22 | 2026-03-22 | 進行中 |

## 目的

1. チャンクサイズの異なる複数コレクションを Firestore 上に並存させ、CLI から並列 Ingest できるようにする
2. Tuning 画面を「Operations Monitor」に改修し、複数ジョブの進捗監視 + コレクション切替を実現する
3. チャット UI からコレクションを切り替えて、体感で回答品質を比較できるようにする

## 背景・課題

### 現状の問題

- チャンクサイズ実験（DD-019-9）では、1つの `chunks` コレクションを使い回して順次実行するため、**実験が直列でしか回せない**
- 実験後にチャットで回答品質を体感確認したくても、再 Ingest しないと別パラメータに切り替えられない
- スコアだけでは判断が難しいケース（「スコアは低いが実用上は十分」等）を検証する手段がない

### Tuning画面の現状と課題

現在のTuning画面（`ui/src/admin/Tuning.tsx`）は以下の機能を持つ:
- パラメータ編集（chunk_size, top_k 等）
- Technique Toggles（ヘッダーインジェクション等）
- Ingest / Evaluate / Re-tune ボタン
- 進捗バー（1ジョブのみ）

**問題点:**
- パラメータ編集・Technique Toggles → `config.py` やCLI引数で十分。UIで変えると「今どっちが正？」になる
- Re-tune → CLIスクリプトの方が並列対応しやすい
- 進捗バー → 1ジョブ前提で、複数コレクションの並列Ingestに対応できない
- タスクステータスが `task_id="ingest"` の1件固定（`src/task_status.py`）

### DD-019-9 実験結果（確定）

| パラメータ | chunks | Overall | semantic | noise_resistance | cross_category |
|-----------|--------|---------|----------|-----------------|----------------|
| baseline (800/150) | 403 | **79.7%** | 58% | 50% | 100% |
| A (600/100) | 593 | **82.4%** | 58% | 50% | 80% |
| B (1200/200) | 292 | **85.1%** | 67% | 67% | 100% |

**傾向:** チャンクが大きいほどスコアが高い。1200が最良だが、さらに大きくすると改善するか、頭打ちかは不明。

## 検討内容

### チャンクサイズの提案

DD-019-9 の結果から、**大きいチャンクが有利**という傾向が明確。以下の3コレクションを提案:

| コレクション | chunk_size | chunk_overlap | 狙い |
|-------------|-----------|--------------|------|
| `chunks_1200` | 1200 | 200 | DD-019-9 の最良結果を保持 |
| `chunks_1600` | 1600 | 300 | さらに大きくして改善幅を探る |
| `chunks_2000` | 2000 | 400 | 上限探索（頭打ちポイントの特定） |

**理由:**
- 600→800→1200 と大きくするほどスコアが上がっている（79.7→82.4は逆転だが、semantic以外はチャンク大が有利）
- contextual_retrieval=ON なので、大チャンクでも検索精度が落ちにくい
- 1200 は確定データがあるので比較基準になる
- 2000 を試すことで「どこで頭打ちか」が分かる

### 影響範囲

`config.collection_name` を参照している箇所:

| ファイル | 用途 |
|---------|------|
| `src/search/retriever.py:39` | ベクトル検索 |
| `src/search/keyword_searcher.py:29` | キーワード検索 |
| `src/ingest/store.py:30,76` | チャンク保存・クリア |
| `src/browse/exporter.py:30` | データブラウザ |
| `main.py:585` | チャット検索 |
| `scripts/ingest.py:25` | Ingest スクリプト |

すべて `config.collection_name` を間接参照しているため、**Config の値を変えるだけで全コードが追従する**。

### 設計案

#### 1. コレクション命名規則

```
chunks              ← 本番（デフォルト）
chunks_1200         ← chunk_size=1200
chunks_1600         ← chunk_size=1600
chunks_2000         ← chunk_size=2000
```

#### 2. Config 変更

`src/config.py` の `collection_name` を環境変数対応にする:
```python
collection_name: str = os.environ.get("COLLECTION_NAME", "chunks")
```

#### 3. タスクステータスの複数対応

`src/task_status.py` の `task_id` をコレクション別にする:
- 現状: `task_id = "ingest"` （1件）
- 変更後: `task_id = "ingest_chunks_1200"` （コレクション別）

Evaluate も同様: `task_id = "evaluate_chunks_1200"`

#### 4. Tuning画面 → Operations Monitor に改修

**廃止する機能:**
- パラメータ編集（chunk_size, top_k 等） → `config.py` / CLI引数
- Technique Toggles → `config.py`
- Re-tune（一括実行） → `scripts/run_chunk_experiments.py`

**新設する機能:**
- コレクション選択ドロップダウン（チャットの検索対象切替）
- Ingest 複数ジョブの進捗一覧（コレクションごとのプログレスバー）
- Evaluate 複数ジョブの進捗一覧
- スコア比較テーブル（完了済みコレクション横断）

**HTMLモック:** `doc/DD/DD-019-9-2/mock_operations_monitor.html`

#### 5. 実験スクリプトの改修

`scripts/run_chunk_experiments.py` を改修:
- 各実験で `config.collection_name` を `chunks_{chunk_size}` に設定
- `task_id` もコレクション別に設定（UIから進捗監視可能にする）
- 既存コレクション `chunks` は触らない

## 決定事項

- チャンクサイズは 1200 / 1600 / 2000 の3パターンで実験（DD-019-9 の結果に基づく）
- Tuning画面はOperations Monitorに改修（パラメータ編集・Re-tuneは廃止、CLIに一本化）
- 実行トリガーはCLI、進捗モニタリングとコレクション切替はUI
