## 目的

顧客が過去に構築したRAGシステムは精度不足でダメ出しされた。本プロジェクトは「なぜRAGの精度が悪くなるのか」「どうすれば精度が上がるのか」を調査し、それを反映したPoCを構築している。この調査・設計内容を、非エンジニア（現場スタッフ）にも伝わるプレゼン資料としてまとめる。

## 背景・課題

- 顧客は過去にRAGシステムを構築したが、精度が悪く実用に至らなかった（構築方法の詳細は不明）
- 本プロジェクトでは `doc/research/` に全8回＋補足のリサーチを実施し、精度を上げるための技術要素を特定済み
- これらの調査内容を非エンジニアに伝え、「今度こそいける」という信頼を得る必要がある
- DD-010で一般的なプロジェクト紹介を作成したが、「精度改善」という本質的なストーリーが弱かったため方針転換

### RAG精度が悪くなる3つの原因（概要編の核）

research資料から抽出した、精度劣化の主要因:

| 原因 | 説明 | 対策 |
|------|------|------|
| **①文書の切り方が悪い** | 意味の途中でぶつ切り → AIが文脈を失う | セマンティックチャンキング、ヘッダーインジェクション |
| **②検索が雑** | キーワード一致だけ → 言い換え・類義語に対応できない | ベクトル検索、クエリ拡張、ハイブリッド検索 |
| **③AIに渡す情報の質が低い** | 関係ない文書まで渡す → AIが混乱して嘘をつく | リランキング、スコア閾値、動的コンテキスト調整 |

### PoCの紹介（概要編セクション5の素材）

現在動いているPoCの構成を、非エンジニアにも分かる形で紹介する。

**PoCで実際にできること:**
- ブラウザでチャット画面を開き、社内文書に関する質問ができる（Streamlit）
- AIが社内文書を検索し、根拠付きで回答する
- 回答の下に「どの文書を参照したか」が表示される
- 45件のテスト質問で自動的に精度を測定できる

**PoCの技術構成（非エンジニア向け説明）:**

| 部品 | 役割 | 具体的には |
|------|------|-----------|
| チャット画面 | ユーザーが質問する入口 | Streamlit（Pythonで作るWebアプリ） |
| 文書データベース | 社内文書を保管・検索 | Google Cloud Firestore |
| 意味変換エンジン | 文書と質問を「意味の数値」に変換 | Vertex AI Embedding |
| AI審査員 | 検索結果の関連度を再評価 | Vertex AI Ranking API |
| 回答生成AI | 文書を読んで回答を作成 | Gemini 2.5 Flash |
| 自動テスト | 45問で精度を自動測定 | 自作の評価パイプライン |

### アーキテクチャ: PoC → 本番拡張（概要編セクション7の素材）

PoCはシンプルな構成だが、本番化に必要な拡張ポイントが明確になっている。

**PoC構成（現在）:**

```
ユーザー → Streamlit（ローカル）→ GCP APIs
                                    ├── Firestore（文書DB）
                                    ├── Vertex AI Embedding（意味変換）
                                    ├── Vertex AI Ranking API（再評価）
                                    └── Gemini 2.5 Flash（回答生成）
```

**本番構成（拡張イメージ）:**

```
ユーザー → Cloud Run（本番UI）→ GCP APIs
  │           │                    ├── Firestore（文書DB + セッション管理）
  │           │                    ├── Vertex AI Embedding
  │           │                    ├── Vertex AI Ranking API
  │           │                    ├── Gemini（Flash / Pro 使い分け）
  │           │                    └── Cloud Storage（元ファイル保管）
  │           │
  │           ├── Firebase Auth（ログイン・権限管理）
  │           ├── Eventarc（文書アップロード検知→自動取り込み）
  │           └── Cloud Monitoring（精度・レイテンシ監視）
  │
  └── 管理者 → 文書アップロード（GCS）→ 自動でインデックス更新
```

**PoC → 本番で何が変わるか:**

| 観点 | PoC（現在） | 本番（拡張後） |
|------|------------|--------------|
| **UI** | Streamlit（ローカル起動） | Cloud Run上のWebアプリ（常時稼働） |
| **同時利用** | 数人 | Cloud Runスケーリングで数百人対応 |
| **認証** | なし（誰でもアクセス可） | Firebase Auth（部署・役職で権限制御） |
| **文書更新** | 手動で `scripts/ingest.py` 実行 | GCSにファイル置くだけで自動反映 |
| **LLMモデル** | Gemini Flash固定 | 質問の難易度に応じてFlash/Pro自動切り替え |
| **監視** | CLI出力のみ | Cloud Monitoringでダッシュボード化 |
| **会話履歴** | ブラウザ閉じたら消える | Firestoreに永続保存 |
| **フィードバック** | なし | 👍👎ボタン→改善キューに自動登録 |

**ポイント: PoCと本番でコア部分（検索・リランキング・回答生成）は同じ。変わるのは周辺の「運用の仕組み」。**
→ PoCで精度が出れば、本番化は周辺を足すだけ。
