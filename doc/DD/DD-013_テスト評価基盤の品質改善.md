# DD-013: テスト評価基盤の品質改善

| 作成日 | 更新日 | ステータス |
|--------|--------|------------|
| 2026-03-21 | 2026-03-21 | 進行中 |

## 目的

現在の自動評価パイプライン（74件のテストケース × キーワード判定）の信頼性を検証し、改善する。テスト結果に基づいてチューニング判断を行うためには、テスト自体の品質が担保されていなければならない。

## 背景・課題

DD-009-4 で管理者向けダッシュボードを構築し、評価スコアを可視化した。しかし表示されるスコアを見て「この数字は信頼できるのか？」という根本的な疑問が浮上した。

### 課題1: テストケース件数の偏り

| パターン | 件数 | 1件あたりのスコア変動 |
|---------|------|---------------------|
| semantic | 10 | ±10% |
| exact_match / step_sequence / multi_chunk / unanswerable | 各5 | ±20% |
| **similar_number / ambiguous / cross_category / security / noise_resistance** | **各3** | **±33%** |

3件しかないパターンでは、1件の成否でスコアが33%振れる。ambiguous 0%、security 0% が「壊滅的」に見えるが、統計的には1-2件の成否で印象が全く変わるレベル。

### 課題2: 未実装機能のテストが混在

| パターン | 現在のスコア | 実態 |
|---------|------------|------|
| ambiguous (曖昧質問) | 0% (0/3) | プロンプトに聞き返し指示がない。07の技術が未実装なので0%は当然 |
| security (権限制御) | 0% (0/3) | 権限フィルタ未実装。08の技術がないので拒否できるわけがない |

これらは「テスト不合格」ではなく「テスト対象の機能がまだ存在しない」。同列にスコア表示すると誤解を招く。

### 課題3: キーワード判定の限界

現在のスコアリング:
```python
score = 含まれていたキーワード数 / 全キーワード数
```

| 問題 | 具体例 |
|------|--------|
| **偽陽性**: キーワードが含まれているが文脈が違う | 「メモリ」が回答に含まれていても、メモリ確認の手順ではない場合がある |
| **偽陰性**: 意味的に正しいが表現が違う | 「アクセス権がありません」→ expected は「権限」。意味は同じだが不合格 |
| **正解が1パターンしかない** | unanswerable-005: 「FAQ-016に誘導しても可」とnoteにあるが、expected_keywordsは「見つかりません」系のみ |

### 課題4: テストケースの正確性

- `unanswerable-005`（Pythonのインストール方法）: noteに「FAQ-016に誘導しても可」とあるが、expected_keywords は拒否パターンのみ。正解が2通りあるのに片方しか認めていない
- `ambiguous` 3件: 聞き返しの expected_keywords に「VPN」「Wi-Fi」が含まれる。回答にこれらが含まれれば文脈無関係に部分スコアが入る
- security 3件: expected_keywords は「権限」「アクセス」等だが、回答が「この情報は社内文書に記載がありません」でも意味的には適切な場合がある

## 検討内容

### 改善の方向性（4軸）

#### A. テストケース件数の正規化

**目標**: 全パターン最低5件、可能なら各パターン均等に

| 対応 | 内容 | 効果 |
|------|------|------|
| 3件のパターンに2件ずつ追加 | similar_number, ambiguous, cross_category, security, noise_resistance | 1件あたりの変動が ±33% → ±20% に改善 |
| 全パターン7件に統一 | 45件 → 70件 | パターン間の比較が公平に。ただし評価実行時間が1.5倍 |

**トレードオフ**: 件数を増やすと評価の実行時間が増加（現在45件で数分 → 70件で約5分）。PoCでは許容範囲だが、追加するテストケースの品質が低いと逆効果。

#### B. 未実装機能テストの分離表示

**選択肢**:

| 案 | 内容 | メリット | デメリット |
|----|------|---------|-----------|
| B1: ラベル分離 | テストケースに `requires` フィールド追加。未実装技術を要求するケースはダッシュボードで「未実装」と表示 | 既存テストケースを削除しない。将来実装時にそのまま使える | テストデータの構造変更が必要 |
| B2: スコア除外 | 未実装機能のテストを Overall スコアから除外し、別セクションに表示 | Overall スコアが「現在の実力」を正確に反映 | 何を除外するかの判断が主観的 |
| B3: 両方表示 | 「実装済みスコア」と「全体スコア」の2つを表示 | 最も情報量が多い | UIが複雑になる |

#### C. スコアリング方法の改善

| 案 | 内容 | メリット | デメリット |
|----|------|---------|-----------|
| C1: キーワード判定の改善 | 同義語リスト、部分一致の緩和、複数正解パターン | 実装が簡単。既存パイプラインの延長 | 根本的な限界は変わらない |
| C2: LLM評価の導入（09の技術） | Geminiに「期待回答と実際の回答の意味的一致度」を0-10で評価させる | 表現の揺れに強い。人間の判断に近い | コスト増（1ケースあたり1 LLM呼び出し追加）。LLM自体の評価が不安定な場合がある |
| C3: ハイブリッド判定 | キーワード判定 + LLM判定の併用。両方のスコアを記録し、乖離が大きいケースをフラグ | 信頼性が最も高い。判定の根拠が2系統 | 実装と実行コストが最大 |

#### D. テストケースの品質レビュー

既存45件を全件レビューし、以下を修正:
- expected_keywords の妥当性（偽陽性/偽陰性を生む曖昧なキーワードの修正）
- 複数正解パターンへの対応
- notes と expected の整合性チェック

DD-013作成時（45件/30チャンク）→ DD-014完了後（74件/454チャンク）で前提が大きく変化。

| 軸 | 当初の課題 | DD-014後の状況 | 優先度変化 |

| **C: スコアリング** | キーワードマッチの限界 | DD-014-2アブレーションで±0〜1.3%しか差が出ず。キーワードマッチの粗さが根本原因の可能性 | **↑ 最優先** — LLM-as-Judge導入が急務 |
| **D: 品質レビュー** | 45件の品質チェック | DD-014で29件追加時に一部修正済み。74件全体の再レビューは未実施 | **→ 中** — C完了後に再評価 |

### 推奨フェーズ順序の変更

**当初**: D(品質レビュー) → B(分離) → A(均等化) → C(スコアリング)
**変更後**: C(スコアリング) → B(分離) → D(品質レビュー) → A(均等化は必要時のみ)

**理由**: DD-014-2のアブレーション結果から、キーワードマッチの粗さが技術貢献度の測定を妨げていることが判明。スコアリング改善なしに他の改善を行っても効果測定が困難。

## 決定事項

- **C: スコアリング方式 → C3（ハイブリッド）、LLM主・キーワード参考**
  - Overall scoreはLLM-as-Judge判定をベースとする
  - キーワード判定は並行実行し、乖離検出・Phase 3品質レビューの入力として活用
  - 乖離パターン（keyword PASS/LLM FAIL、keyword FAIL/LLM PASS）を自動フラグ
- **B: 未実装テスト分離 → B1拡張（requiresフィールド + 機能ON/OFF連動）**
  - テストケースに `requires` フィールドを追加（例: `"requires": "permission_filter"`）
  - 評価時に機能のON/OFF状態と照合し、OFFまたは未実装の機能を要求するテストはスコアから除外
  - 未実装→実装OFF→実装ONでスコアが自然に変動する仕組み（アブレーションとも連動）
- **D: 品質レビュー → 乖離ケース駆動レビュー**
  - 74件全件を目視ではなく、Phase 1のLLM-as-Judge導入後にキーワード/LLM判定が乖離したケースを重点レビュー
- **A: 件数均等化 → 積極的追加はしない**
  - DD-014でほぼ解決済み（temporal 3件のみ少ないが許容）。Phase 3で不足が判明した場合のみ追加
