# DD-022: コレクション切替による並列チャンク実験

| 作成日 | 更新日 | ステータス |
|--------|--------|------------|
| 2026-03-22 | 2026-03-22 | Phase 0完了 |

## 目的

1. チャンクサイズの異なる複数コレクションを Firestore 上に並存させ、CLI から並列 Ingest できるようにする
2. Tuning 画面を「Operations Monitor」に改修し、複数ジョブの進捗監視 + コレクション切替を実現する

## 背景・課題

### 現状の問題

- チャンクサイズ実験（DD-019-9）では、1つの `chunks` コレクションを使い回して順次実行するため、**実験が直列でしか回せない**
- 実験後にチャットで回答品質を体感確認したくても、再 Ingest しないと別パラメータに切り替えられない
- スコアだけでは判断が難しいケース（「スコアは低いが実用上は十分」等）を検証する手段がない

### Tuning画面の現状と課題

現在のTuning画面（`ui/src/admin/Tuning.tsx`）は以下の機能を持つ:
- パラメータ編集（chunk_size, top_k 等）
- Technique Toggles（ヘッダーインジェクション等）
- Ingest / Evaluate / Re-tune ボタン
- 進捗バー（1ジョブのみ）

**問題点:**
- パラメータ編集・Technique Toggles → `config.py` やCLI引数で十分。UIで変えると「今どっちが正？」になる
- Re-tune → CLIスクリプトの方が並列対応しやすい
- 進捗バー → 1ジョブ前提で、複数コレクションの並列Ingestに対応できない
- タスクステータスが `task_id="ingest"` の1件固定（`src/task_status.py`）

### DD-019-9 実験結果（確定）

| パラメータ | chunks | Overall | semantic | noise_resistance | cross_category |
|-----------|--------|---------|----------|-----------------|----------------|
| baseline (800/150) | 403 | **79.7%** | 58% | 50% | 100% |
| A (600/100) | 593 | **82.4%** | 58% | 50% | 80% |
| B (1200/200) | 292 | **85.1%** | 67% | 67% | 100% |

**傾向:** チャンクが大きいほどスコアが高い。1200が最良だが、さらに大きくすると改善するか、頭打ちかは不明。

## 検討内容

### チャンクサイズの提案

DD-019-9 の結果から、**大きいチャンクが有利**という傾向が明確。以下の3コレクションを提案:

| コレクション | chunk_size | chunk_overlap | 狙い |
|-------------|-----------|--------------|------|
| `chunks_1200` | 1200 | 200 | DD-019-9 の最良結果を保持 |
| `chunks_1600` | 1600 | 300 | さらに大きくして改善幅を探る |
| `chunks_2000` | 2000 | 400 | 上限探索（頭打ちポイントの特定） |

**理由:**
- 600→800→1200 と大きくするほどスコアが上がっている（79.7→82.4は逆転だが、semantic以外はチャンク大が有利）
- contextual_retrieval=ON なので、大チャンクでも検索精度が落ちにくい
- 1200 は確定データがあるので比較基準になる
- 2000 を試すことで「どこで頭打ちか」が分かる

### 影響範囲

`config.collection_name` を参照している箇所:

| ファイル | 用途 |
|---------|------|
| `src/search/retriever.py:39` | ベクトル検索 |
| `src/search/keyword_searcher.py:29` | キーワード検索 |
| `src/ingest/store.py:30,76` | チャンク保存・クリア |
| `src/browse/exporter.py:30` | データブラウザ |
| `main.py:585` | チャット検索 |
| `scripts/ingest.py:25` | Ingest スクリプト |

すべて `config.collection_name` を間接参照しているため、**Config の値を変えるだけで全コードが追従する**。

### 設計案

#### 1. コレクション命名規則

```
chunks              ← 本番（デフォルト）
chunks_1200         ← chunk_size=1200
chunks_1600         ← chunk_size=1600
chunks_2000         ← chunk_size=2000
```

#### 2. Config 変更

`src/config.py` の `collection_name` を環境変数対応にする:
```python
collection_name: str = os.environ.get("COLLECTION_NAME", "chunks")
```

#### 3. タスクステータスの複数対応

`src/task_status.py` の `task_id` をコレクション別にする:
- 現状: `task_id = "ingest"` （1件固定）
- 変更後: `task_id = "ingest:chunks_1200"` （コレクション別）

Evaluate も同様: `task_id = "evaluate:chunks_1200"`

**命名規則:** `{操作}:{コレクション名}` — コロン区切りで操作とコレクションを分離。一覧取得時にプレフィクスフィルタしやすい。

**新規関数: `list_tasks(prefix)`**

```python
def list_tasks(prefix: str = "") -> list[dict]:
    """task_status コレクションからプレフィクスに一致するドキュメントを一覧取得する。

    Args:
        prefix: タスクIDのプレフィクス（例: "ingest:" → ingest系全部）
                空文字なら全件。
    Returns:
        [{"task_id": "ingest:chunks_1200", "running": True, "current": 3, ...}, ...]
    """
```

実装: Firestoreは前方一致クエリが可能（`>=` prefix, `<` prefix + '\uffff'`）。
`task_status` コレクションのドキュメント数は高々 10 件程度なので、全件取得してフィルタでも問題ない。
シンプルに全件取得 + Pythonフィルタを採用する。

#### 4. API エンドポイント設計

既存の admin API パスベースルーティング（`main.py` `admin()`）に以下を追加:

##### GET /collections — コレクション一覧

**取得方法:** Firestore の `chunks*` コレクションをリストアップする。
Firestore Admin SDK の `db.collections()` で全トップレベルコレクションを取得し、`chunks` プレフィクスでフィルタ。
各コレクションのチャンク数は `collection.count().get()` で取得（集計クエリ、読み取り課金なし）。

**評価スコアの取得:** 既存の `eval_results` コレクション（Firestore）にはコレクション名が記録されていない。
→ 評価結果の `config_params.collection_name`（本DDで追加）から逆引きする。
→ 未評価のコレクションは `score: null` を返す。

```
GET /api/admin/collections

Response 200:
{
  "collections": [
    {
      "name": "chunks",
      "chunk_count": 403,
      "score": { "overall": 0.797, "date": "2026-03-20" }   // eval_results から
    },
    {
      "name": "chunks_1200",
      "chunk_count": 292,
      "score": { "overall": 0.851, "date": "2026-03-21" }
    },
    {
      "name": "chunks_1600",
      "chunk_count": 0,
      "score": null                                          // 未評価
    }
  ],
  "active": "chunks"       // 現在の config.collection_name
}
```

##### PUT /collections/active — アクティブコレクション切替

既存の `PUT /config` を拡張する案もあるが、**意味が異なる**ため別エンドポイントにする:
- `PUT /config` = チューニングパラメータの変更（検索閾値等）
- `PUT /collections/active` = どのデータセットを使うかの切替

```
PUT /api/admin/collections/active
Body: { "name": "chunks_1200" }

Response 200:
{ "active": "chunks_1200", "previous": "chunks" }

Response 400:  （存在しないコレクション名の場合）
{ "error": "Collection 'chunks_9999' not found" }
```

**実装:** `config.collection_name` を書き換える。ランタイムのみ（再デプロイでリセット）。
既存の `PUT /config` と同じパターン。バリデーションとして `db.collections()` でコレクション存在確認を行う。

##### GET /tasks — タスク一覧

```
GET /api/admin/tasks
GET /api/admin/tasks?prefix=ingest:

Response 200:
{
  "tasks": [
    {
      "task_id": "ingest:chunks_1200",
      "running": true,
      "current": 5,
      "total": 12,
      "current_file": "guide/setup.md",
      "elapsed": 45.2,
      "estimated_remaining": 63.0
    },
    {
      "task_id": "ingest:chunks_1600",
      "running": false
    }
  ]
}
```

#### 7. Tuning画面 → Operations Monitor に改修

**廃止する機能:**
- パラメータ編集（chunk_size, top_k 等） → `config.py` / CLI引数
- Technique Toggles → `config.py`
- Re-tune（一括実行） → `scripts/run_chunk_experiments.py`

**新設する機能:**
- コレクション選択ドロップダウン（チャンク数・評価スコア付き）
- Ingest / Evaluate 複数ジョブの進捗一覧（コレクションごとのプログレスバー）
- スコア比較テーブル（完了済みコレクション横断）

**HTMLモック:** `doc/DD/DD-022/mock_operations_monitor.html`

> **方針変更（2026-03-22）:** コレクション選択はチャット画面ではなくOperations Monitorに配置。チャット画面はシンプルに保ち、設定変更は管理画面に集約する。DD-021（検索プリセット）破棄と同じ方針 — チャット画面にユーザーが判断しづらい切替UIを置かない。

**UI → API マッピング:**

| UI操作 | API呼び出し | タイミング |
|--------|-----------|-----------|
| 画面オープン | `GET /collections` | マウント時 |
| 画面オープン | `GET /tasks` | マウント時 + 2秒ポーリング |
| コレクション切替 | `PUT /collections/active` | ドロップダウン変更時 |
| ジョブキャンセル | `POST /ingest/cancel` or `POST /evaluate/cancel` | キャンセルボタン |

**既存UIからの削除対象（`Tuning.tsx`）:**
- `PARAM_FIELDS` 定義と Parameters セクション全体（L210-311）
- Technique Toggles セクション全体（L243-284）
- `handleRetune()` と Re-tune ボタン（L181-203, L413-420）
- `handleSaveConfig()` と Save Parameters ボタン（L125-135, L304-311）

**既存UIから保持する要素:**
- Ingest / Evaluate の実行ボタン・進捗バー・キャンセル（→ 複数ジョブ対応に改修）
- Evaluation Result 表示（→ スコア比較テーブルに発展）

#### 8. 実験スクリプトの改修

`scripts/run_chunk_experiments.py` を改修:
- 各実験で `config.collection_name` を `chunks_{chunk_size}` に設定
- `task_id` を `f"ingest:chunks_{chunk_size}"` に設定（UIから進捗監視可能にする）
- 既存コレクション `chunks` は触らない

`scripts/ingest.py` に CLI 引数を追加:
```
python scripts/ingest.py --chunk-size 1200 --chunk-overlap 200 --collection chunks_1200 --clear
```

引数なしの場合は既存動作（`config.py` のデフォルト値）を維持。

## 決定事項

- チャンクサイズは 1200 / 1600 / 2000 の3パターンで実験（DD-019-9 の結果に基づく）
- Tuning画面はOperations Monitorに改修（パラメータ編集・Re-tuneは廃止、CLIに一本化）
- コレクション選択はOperations Monitorに配置（チャット画面には置かない）
- 実行トリガーはCLI、進捗モニタリングとコレクション切替は管理画面
