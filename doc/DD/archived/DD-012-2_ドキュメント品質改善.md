## 目的

doc/spec/ のドキュメント哲学を転換する。

**原則: 人間が管理すべきは抽象的な概念と設計意図。具体的なコードはLLMに任せればよい。**

コードを転写したドキュメント（フィールド定義、JSONサンプル、UIパーツ一覧）は「コードが変わったら陳腐化する負債」でしかない。人間が読んで理解が深まるドキュメント ＝ コードからは読み取れない「なぜ・どう考えたか・全体の流れ」を記述したもの。

## 背景・課題

DD-012で作成した doc/spec/ は、コードの転写に終始しており人間の理解を深めない：

- **screens.md**（234行）: UIコンポーネント・テーブルカラム・CSSクラスの羅列 → コードを読めば全部分かる
- **api.md**（390行）: JSONサンプルとフィールド定義の転写 → `main.py` を読めば全部分かる
- **database.md**: Firestoreフィールド定義の転写 → コードが正。ドキュメントは劣化コピー
- **infrastructure.md**: サービス一覧は良いが、開発 vs 本番の構成差異が見えない

**問題の本質**: 「何があるか」は書いてあるが「なぜそうなっているか」「全体がどう繋がるか」が書かれていない。

## 決定事項

### ドキュメント哲学

| 書くべき（人間の仕事） | 書かないべき（LLMが読めばいい） |
|----------------------|--------------------------|
| 設計意図・Why | フィールド定義・型 |
| 概念モデル・用語定義 | JSONサンプル |
| データの流れ・パイプライン構造 | UIコンポーネント詳細 |
| 設計判断とトレードオフ | パラメータ一覧 |
| 開発 vs 本番の差異 | CSSクラス・テーブルカラム |
| ユーザーの操作フロー・意思決定 | API リクエスト/レスポンス定義 |

### 各ファイルの方針

#### screens.md → 「画面設計と操作フロー」
- **削除**: UIパーツ詳細、テーブルカラム、API対応の羅列
- **書く**: 各画面の存在目的、ユーザーの操作フロー（チューニングサイクル）、画面間の関係図

#### api.md → 「API設計思想」
- **削除**: 全エンドポイントのJSON定義（390行→大幅圧縮）
- **書く**: APIの設計思想（なぜ2関数構成か、なぜpath-basedルーティングか）、RAGパイプラインのデータフロー、エンドポイント一覧表（目的ベース、詳細なし）

#### database.md → 「データモデル概念」
- **削除**: フィールド単位の型定義
- **書く**: コレクションの役割と関係（概念ER図）、データのライフサイクル（Ingest→検索→ログ→評価の流れ）、設計判断（なぜFirestoreか、なぜベクトルDBを兼用か）

#### infrastructure.md → 「開発/本番アーキテクチャ」
- **追加**: 開発環境構成（Vite + functions-framework）、本番環境構成（DD-016完了前提）、差分対比
- **既存のサービス一覧**: 維持（これは概要として価値がある）
