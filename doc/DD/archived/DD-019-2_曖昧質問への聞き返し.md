# DD-019-2: 曖昧質問への聞き返し

| 作成日 | 更新日 | ステータス |

## 目的

曖昧な質問（「あれの設定方法を教えて」「エラーが出る」等）に対して、RAGパイプラインが回答を推測せずに聞き返しを返す機能を実装する。ambiguousテスト 0/5 → 改善を目指す。

## 背景・課題

現在のRAGパイプラインは全ての入力をそのままベクトル検索にかけ、関連度の低い文書でも無理やり回答を生成してしまう。曖昧な質問では的外れな回答になりやすく、ユーザー体験を損なう。

**現状（ambiguousテスト: 0/5）:**
- `ambiguous-001`: 「あれの設定方法を教えて」→ 無関係な文書で回答生成
- `ambiguous-002`: 「接続できない」→ 推測で特定の手順を回答
- 他3件も同様

**テストデータ**（`test-data/golden/eval_dataset.jsonl`）に5件の評価ケースが既存。全て `"requires": "clarification"` で現在スキップされている。

### 技術選定: LLMベース判定 vs ルールベース判定

| 方式 | メリット | デメリット |
|------|---------|-----------|
| LLMベース（Gemini呼び出し） | 高精度、文脈理解可能 | レイテンシ増（1-2秒）、API費用 |
| ルールベース（キーワード/長さ） | 高速、無料 | 誤検知多い、拡張困難 |

**→ LLMベースを採用。** 理由: 曖昧さの判定は文脈依存が高く、ルールベースでは「申請したい」が曖昧で「VPN申請したい」が明確、といった区別が困難。Gemini 2.5 Flashのコストは十分低く、PoC段階ではレイテンシより精度を優先する。

## 検討内容

### 実装方式: Pre-RAG判定

```
Query → [曖昧判定(LLM)] → 曖昧: 聞き返し応答を返す（検索スキップ）
                          → 明確: 通常のRAGフロー
```

**Pre-RAG判定を選択した理由:**
- 曖昧な質問で不要なベクトル検索・リランキングを省ける
- レスポンス構造がシンプル（聞き返し or 通常回答の二択）
- 検索結果に依存しない判定のため、検索品質に左右されない

### 曖昧判定プロンプト設計

Geminiに以下を判定させる:
1. **is_ambiguous** (bool): 質問が曖昧で聞き返しが必要か
2. **clarification_question** (str): 聞き返し文（曖昧な場合のみ）

判定基準:
- 対象が不明確（「あれ」「それ」等の指示語のみ）
- 状況の説明が不足（「エラーが出る」→何のエラー？）
- 複数の解釈が可能（「申請したい」→何の申請？）

非曖昧と判定すべきケース:
- 具体的な対象がある（「VPNの設定方法を教えて」）
- 十分な文脈がある（「ExcelでVLOOKUPのエラーが出る」）

### レスポンス拡張

`rag_flow()` の戻り値に `is_clarification` フラグを追加:

```python
# 通常回答
{"answer": "...", "sources": [...], "is_clarification": False}

# 聞き返し
{"answer": "何の設定方法について...", "sources": [], "is_clarification": True}
```

### Config統合

```python
# src/config.py に追加
clarification: bool = True

# src/evaluate/scorer.py FEATURE_MAP 更新
"clarification": "clarification",  # None → "clarification"
```
