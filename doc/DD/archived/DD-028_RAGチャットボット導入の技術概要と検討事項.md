# DD-028: RAGチャットボット導入の技術概要と検討事項

| 作成日 | 更新日 | ステータス |
|--------|--------|------------|
| 2026-03-25 | 2026-03-25 | 完了 |

## 目的

チャットボット導入の打診を受け、初回ミーティングで以下を提示する：
- チャットボットの構成を理解していること
- Vertex AI Searchをベースに構築する方針と、その理由
- チャットボットとして実装する際に注意すべきポイント

## 背景・課題

- 社内文書検索のRAGチャットボットPoCを完了済み（評価スコア86%）
- PoCでは自前RAG（Firestore + Gemini）を構築したが、検索精度の維持・運用に手間がかかることを実感
- **本番はVertex AI Searchに検索を任せ、精度向上はデータ加工で対応する方針**
- チャットボット導入を打診された方（ビジネス寄り）に、技術理解と方針を示す必要がある

## 検討内容

### 1. チャットボットの全体構成

```
┌─────────────────────────────────┐
│  ユーザー（ブラウザ / Slack等）    │
└──────────────┬──────────────────┘
               │ 質問
               ▼
┌─────────────────────────────────┐
│  チャットボット（自社実装部分）     │
│  ・会話の管理                     │
│  ・回答の表示・出典の提示          │
│  ・認証・権限                     │
└──────────────┬──────────────────┘
               │ 検索リクエスト
               ▼
┌─────────────────────────────────┐
│  Vertex AI Search（GCPサービス）  │
│  ・文書の取込・索引付け            │
│  ・検索（意味検索＋キーワード検索） │
│  ・回答生成（Grounding付き）       │
└─────────────────────────────────┘
```

**ポイント：**
- 検索・回答生成の重い部分はVertex AI Searchが担う（GCPマネージドサービス）
- 自社で実装するのは「チャットボットとしての体験」の部分

### 2. なぜVertex AI Searchを使うのか

| 観点 | 自前RAG（PoCで経験） | Vertex AI Search |
|------|---------------------|-----------------|
| 検索精度の調整 | チャンクサイズ、検索アルゴリズム等を自分で最適化する必要がある | GCP側が最適化してくれる。精度向上はデータ加工で対応 |
| 運用負荷 | ベクトルDB管理、再Ingest、インデックス更新を自前で行う | 文書をアップロードすれば自動で索引化される |
| スケーラビリティ | 自分でインフラ設計が必要 | GCPが自動スケール |
| 新機能の追従 | 検索技術の進化に自分で追従する必要がある | GCP側で継続的に改善される |

> PoCで自前RAGを一通り構築した結果、**検索部分の運用コストが高い**ことを実感した。
> 本番では検索をVertex AI Searchに委ね、自社のリソースはチャットボット体験の作り込みとデータ品質に集中すべき。

### 3. チャットボット実装で注意すべきポイント

Vertex AI Searchが検索・回答生成を担うとして、**チャットボットとして成立させるために自社で考えるべきこと**：

#### A. データの品質（精度に直結する）

| ポイント | 説明 |
|---------|------|
| **文書の前処理** | PDFやWord等をそのまま入れるのではなく、構造化・クリーニングしてから投入する。見出し・表の構造が壊れると検索精度が下がる |
| **メタデータの付与** | 文書にカテゴリ・部署・更新日等のメタデータを付けると、検索のフィルタリングや回答の精度が上がる |
| **更新運用の設計** | 元文書が更新されたらVertex AI Search側も再取込する仕組みが必要。手動では漏れる |

#### B. ユーザー体験（使ってもらえるかどうかに直結する）

| ポイント | 説明 |
|---------|------|
| **回答の出典表示** | 「どの文書のどこを根拠に回答したか」を示す。信頼性の担保に必須 |
| **回答できない時の振る舞い** | 文書にない質問には「わかりません」と正直に返す。でっち上げ（ハルシネーション）が最大のリスク |
| **会話の文脈保持** | 「それについてもう少し詳しく」等の追加質問に対応できるか。1問1答でも最低限は成立する |
| **利用導線** | Webアプリ / Slack / Teams のどれで使えるか。利用者の普段の業務動線に合わせる |

#### C. セキュリティと権限（社内情報を扱う以上、必須）

| ポイント | 説明 |
|---------|------|
| **認証** | 誰がチャットボットを使えるかを制御する（Google Workspace SSO等） |
| **文書アクセス権限** | 人事文書を全員が検索できてはいけない等、文書ごとの閲覧権限の設計 |
| **ログ・監査** | 誰がどんな質問をしたかのログ。コンプライアンス上も必要になりうる |

#### D. コストと運用

| ポイント | 説明 |
|---------|------|
| **API利用料の見積もり** | Vertex AI Searchの料金体系（クエリ数課金）を利用者数×質問頻度で試算 |
| **段階的リリース** | まず小規模チームで試用→フィードバック→改善のサイクルを回す |
| **品質モニタリング** | 利用者のフィードバック収集と定期的な精度確認の仕組み |

### 4. 提案する進め方

```
Step 1: 要件の明確化
  ・利用者は誰か、対象文書は何か、どこで使うか

Step 2: Vertex AI Searchへのデータ投入・検証
  ・対象文書を整備してVertex AI Searchに投入
  ・検索精度が実用レベルか確認

Step 3: チャットボットのプロトタイプ実装
  ・最小限のUI + Vertex AI Search連携
  ・小規模チームで試用

Step 4: フィードバック→改善→段階的拡大
```
