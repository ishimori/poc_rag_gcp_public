# RAG精度改善テクニックのファクトチェック用プロンプト

## 使い方

以下のプロンプトを他のLLM（Gemini、GPT-4o等）に投げて、各テクニックの有効性を検証してください。
回答を持ち帰って、ステータスマップの記載内容と突き合わせます。

---

## プロンプト

```
あなたはRAG（Retrieval-Augmented Generation）システムの専門家です。
以下は、社内文書Q&AのRAGシステムの精度を改善するために我々が採用を検討している技術の一覧です。

各技術について、以下の観点で評価してください。

【評価観点】
1. 2026年現在、この技術はまだ有効か？（より良い代替手段が登場していないか）
2. 精度改善への寄与度は？（高・中・低）
3. 注意点やリスクはあるか？
4. 他に検討すべき技術で、このリストに抜けているものはあるか？

【前提条件】
- GCP上で構築（Firestore、Vertex AI、Discovery Engine）
- 対象文書: 社内マニュアル、FAQ、部品仕様書、規程類（日本語中心）
- 想定ユーザー: 3,000人規模の企業
- LLM: Gemini 2.5 Flash（必要に応じてPro）
- Embedding: text-embedding-005（768次元）

【検証対象の技術一覧】

■ 文書の分割（チャンキング）
1. RecursiveCharacterTextSplitter（800トークン/150オーバーラップ）
   → 日本語のセパレータ（\n##, \n###, \n\n, \n, 空白）で再帰的に分割
2. ヘッダーインジェクション
   → 各チャンクの先頭に元文書のタイトル・カテゴリを付与し、断片でも文脈が分かるようにする
3. セマンティックチャンキング
   → 文間のベクトル類似度を計算し、意味の切れ目で分割する方式

■ 検索
4. ベクトル検索（Firestore find_nearest, COSINE距離）
   → 質問と文書チャンクを同じEmbeddingモデルで数値化し、意味の近さで検索
5. クエリ拡張（Multi-Query）
   → LLMで1つの質問を3〜5の異なる検索クエリに変換して検索精度を上げる
6. HyDE（Hypothetical Document Embeddings）
   → LLMで仮の回答を生成し、その回答のベクトルで検索する（質問ベクトルより文書ベクトルに近くなる）
7. ハイブリッド検索 + RRF（Reciprocal Rank Fusion）
   → ベクトル検索とキーワード検索を併用し、RRFでスコアを統合する

■ 検索結果の精錬
8. リランキング（Cross-Encoder方式、Vertex AI Ranking API）
   → 検索結果をCross-Encoderで再評価し、本当に関連度の高い文書だけに絞る
9. 動的コンテキスト調整
   → 質問タイプに応じてAIに渡すチャンク数を変える（型番検索=1件、トラブルシュート=7件など）

■ AIの振る舞い
10. インテントルーティング
    → 質問の種類（IT/部品/人事等）をLLMで判定し、検索戦略やプロンプトを切り替える
11. 逆質問（曖昧クエリ対応）
    → 質問の情報が不足しているとき、検索せずに「OSは何ですか？」等と聞き返す
12. ReActループ（多段階推論）
    → Thought→Action→Observationのサイクルで、複数回の検索・推論を繰り返す

■ セキュリティ
13. 事前フィルタリング（Document Level Security）
    → 検索前にユーザーの権限（部署・役職）でFirestoreクエリを絞り込む（Post-filterではなくPre-filter）

■ 評価
14. キーワードベースの自動評価（期待キーワードの含有チェック）
15. RAG Triad評価（Faithfulness, Answer Relevancy, Context Precision）
    → LLMを使って回答品質を多角的に自動判定

【回答フォーマット】
各技術について以下の形式で回答してください：

| # | 技術名 | 2026年現在の有効性 | 精度寄与度 | 注意点・リスク |
|---|--------|-------------------|-----------|---------------|

最後に「リストに抜けている重要な技術」があれば追加してください。
```

---

## 回答の突き合わせ方針

- 「もう古い」と判定された技術 → ステータスマップから削除 or 代替技術に差し替え
- 「抜けている」と指摘された技術 → research資料で調査済みか確認。未調査なら追加調査を検討
- 複数LLMで評価が割れた技術 → 両方の見解を併記して判断材料にする
