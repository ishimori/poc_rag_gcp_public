# LLM相談結果サマリー（2026-03-22）

Gemini 2.5 Pro と GPT-4o に同じプロンプトで相談した結果を整理。

## 両者の共通見解

**「81.1%の時点で検索精度の問題ではなく、制御・判定ロジックの問題」** という診断が一致。
特に security 0% は「RAGの失敗」ではなく「状態伝達の失敗」である点を両者とも指摘。

## カテゴリ別 改善策の比較

### 1. security 0% → 100%（両者: 最優先）

| | Gemini | GPT |
|---|---|---|
| 対策名 | 2段階検索（権限チェック分離） | Access Denied Signal Injection |
| 本質 | **同じ**: フィルタなし検索とフィルタあり検索の差分で「権限除外」を検出し、LLMに明示 |
| 実装 | フィルタなし→メタデータのみ取得→権限判定→LLMに伝達 | フィルタなし/あり検索→差分判定→`retrieval_status`をLLMに注入 |
| GPT追加案 | — | `FILTERED_BY_PERMISSION` 時はLLMを呼ばずアプリ側で即拒否（最も堅牢） |
| 難易度 | 中 | 中 |
| 期待効果 | security 100%, overall +6〜7pt | 同じ |

**結論**: 両者のアプローチはほぼ同一。GPTの「LLMを呼ばずアプリ側で即拒否」が最もシンプルで堅牢。

---

### 2. semantic 67% → 90%（両者: 高優先）

| | Gemini | GPT |
|---|---|---|
| 主な対策 | **HyDE**（仮説文書Embedding） | **Multi-Query Expansion**（クエリ複数展開） |
| 概要 | LLMに仮の回答を生成させ、その回答のベクトルで検索 | LLMでクエリを3〜5個に言い換えて全て検索→RRF統合 |
| GPT追加案 | — | BM25重み最適化、Answer-Aware Retrieval（生成後再検索） |
| 難易度 | 中 | 中 |
| 期待効果 | semantic残り4件の改善 | semantic 67%→90%, overall +4〜6pt |

**結論**: HyDEとMulti-Query Expansionは異なるアプローチだが、両方試す価値あり。GPTのMulti-Queryの方が実装がシンプルで既存RRFに統合しやすい。

---

### 3. ambiguous 60% → 100%

| | Gemini | GPT |
|---|---|---|
| 対策 | Few-Shot + CoT（判定プロンプト改善） | ルールベース併用 + 温度0固定 + JSON出力強制 |
| 概要 | 判定プロンプトに具体例を追加し、思考過程を言語化させる | LLM判定に加え、主語欠落・指示語等のルール判定を併用。温度0で再現性確保 |
| 難易度 | 低 | 低 |

**結論**: 両方を組み合わせるのが最良。温度0 + JSON出力 + Few-Shotで判定を安定化。

---

### 4. unanswerable 80% → 100%

| | Gemini | GPT |
|---|---|---|
| 対策 | 事後検証（Self-Correction / NLI） | Answerability Gate（スコア閾値 + NLI検証） |
| 概要 | 生成後に「コンテキストのみから導出されているか」をチェック | リランキング後のスコアが低ければ強制拒否、中間なら NLI検証 |
| GPT追加案 | — | 3段階実装（閾値→Top-k平均→NLI）、段階的に精度向上 |
| 難易度 | 中 | 低〜中 |

**結論**: GPTの段階的アプローチが実用的。まずスコア閾値ゲートから始め、不足ならNLIを追加。

---

### 5. 90%超えの追加施策

| | Gemini | GPT |
|---|---|---|
| 提案 | セマンティック・チャンキング（構造単位分割） | Top-k動的調整、チャンク再設計、Rerankerに意図情報付与 |
| 難易度 | 中（再Ingest要） | 低〜中 |

---

### 6. アーキテクチャ全体の改善

| | Gemini | GPT |
|---|---|---|
| 提案 | Agentic RAG（ツール呼び出し型）への移行 | 「検索中心型」→「制御中心型RAG」への進化 |
| 概要 | LangGraph/Function Callingでツール選択をLLMに委任 | クエリ理解→動的検索戦略→検索→回答可能性判定→生成→検証の6段階 |
| 推奨時期 | 直列で90%達成後に着手 | 同じ（まず90%を直列で達成） |

**結論**: 両者とも「まず直列パイプラインで90%を達成してから構造変更」を推奨。

---

## 優先順位（ROI順）

両者の提案を統合した推奨実装順序:

| 順位 | 対策 | 対象カテゴリ | 難易度 | 期待効果 |
|------|------|-------------|--------|----------|
| **1** | Access Denied Signal（権限検出→即拒否） | security 0%→100% | 中 | **+6〜7pt** |
| **2** | Multi-Query Expansion（クエリ複数展開） | semantic 67%→85〜90% | 中 | **+4〜6pt** |
| **3** | Answerability Gate（スコア閾値ゲート） | unanswerable 80%→100% | 低 | **+1〜2pt** |
| **4** | 曖昧判定の安定化（温度0 + Few-Shot） | ambiguous 60%→100% | 低 | **+2〜3pt** |
| 5 | Entailment検証（生成後の根拠確認） | 全体の安定化 | 中 | **+2〜3pt** |
| 6 | セマンティック・チャンキング | semantic + table | 中（再Ingest） | **+2〜3pt** |

**上位4つを実装すれば 90〜93% に到達する見込み。**

---

## 判断基準（GPTの提案を採用）

上位4施策を実装後にフル評価を回し:
- 88%未満 → 追加施策（Entailment検証等）が必須
- 88〜91% → 安定化施策を検討
- 92%以上 → PoCとして十分成功
