### 各技術の調査結果

（Phase 1〜4で記入）

#### 1. Contextual Retrieval

**概要:**

Contextual Retrieval は **Anthropic が2024年9月に発表**した RAG の検索精度改善手法。従来の RAG ではドキュメントをチャンクに分割する際に文脈が失われる問題（例:「同社の売上は前四半期比3%増」というチャンクだけでは、どの会社・いつの話か不明）を解決する。

具体的には、**各チャンクをEmbeddingする前に、LLM を使って「そのチャンクがドキュメント全体の中でどういう位置づけか」を説明する短文（50〜100トークン）を生成し、チャンクの先頭に付与する**。これにより、Embedding ベクトルがチャンク単体の意味だけでなく、文書全体の文脈を反映したものになる。

Anthropic の推奨する完全構成は以下の3層:
1. **Contextual Embeddings** — LLM生成の文脈をチャンクに付与してからベクトル化
2. **Contextual BM25** — 同じ文脈付きチャンクでキーワード検索インデックスも構築
3. **Reranking** — 上記2つの検索結果を統合（RRF）した後、リランカーで最終選別

Anthropic が公開しているプロンプトテンプレート:
```
<document>
{{WHOLE_DOCUMENT}}
</document>
Here is the chunk we want to situate within the whole document
<chunk>
{{CHUNK_CONTENT}}
</chunk>
Please give a short succinct context to situate this chunk within the
overall document for the purposes of improving search retrieval of the
chunk. Answer only with the succinct context and nothing else.
```

**ヘッダーインジェクションとの違い:**

| 観点 | ヘッダーインジェクション（実装済み） | Contextual Retrieval |
|------|--------------------------------------|----------------------|
| **付与する情報** | 文書タイトル・カテゴリなど**固定的なメタデータ** | LLMが生成した**チャンク固有の文脈説明文** |
| **生成方法** | ルールベース（タイトル文字列を機械的に結合） | **LLMがドキュメント全体を読んで動的に生成** |
| **粒度** | 同一文書内の全チャンクに同じヘッダー | **チャンクごとに異なる文脈説明**（「この段落は〇〇の仕様のうち△△について述べている」等） |
| **コスト** | ゼロ（文字列結合のみ） | **インデクシング時にLLM呼び出しが必要**（チャンク数 × 1回） |
| **効果** | 基本的だが有効（両者一致で必須級と評価済み） | ヘッダーインジェクションの上位互換。Anthropicは「汎用的な文書要約の付与では効果が限定的」と明記しており、**チャンク固有の文脈生成が鍵** |

要するに、ヘッダーインジェクションは「このチャンクは○○というドキュメントの一部です」という静的ラベル。Contextual Retrieval は「このチャンクはドキュメント全体の中で具体的に△△の文脈で□□を説明している部分です」というLLM生成の動的な文脈付与。**ヘッダーインジェクションの完全な上位互換**と言える。

**精度への寄与度:**

Anthropic の公式ベンチマーク（Top-20チャンク検索失敗率で測定）:

| 手法 | 検索失敗率 | ベースラインからの改善 |
|------|-----------|---------------------|
| ベースライン（通常のEmbedding） | 5.7% | — |
| **Contextual Embeddings のみ** | 3.7% | **35%削減** |
| **Contextual Embeddings + Contextual BM25** | 2.9% | **49%削減** |
| **上記 + Reranking** | 1.9% | **67%削減** |

補足:
- FinanceBench（金融文書QA）では、Contextual Chunk Headers + RSE の組み合わせでスコアが **19% → 83%** に向上した事例も報告されている
- 2026年現在、RAGの検索エンジニアリングのベストプラクティスとして広く採用されており、「ハイブリッド検索 + リランキング + コンテキスト付与」が標準構成とされている

**GCP環境での実現方法:**

本プロジェクトの技術スタック（Gemini + Firestore + Vertex AI Embedding）での実装アプローチ:

**Step 1: インデクシング時のコンテキスト生成**
```
[既存の ingest パイプライン]
  ↓ チャンク分割後
  ↓
[Gemini 2.5 Flash でコンテキスト生成]  ← 追加ステップ
  - 入力: ドキュメント全文 + 個別チャンク
  - 出力: 50〜100トークンの文脈説明文
  - Vertex AI の Context Caching を活用（同一ドキュメント内のチャンク処理でドキュメント全文をキャッシュ）
  ↓
[コンテキスト + 元チャンクを結合]
  ↓
[Vertex AI Embedding API でベクトル化]
  ↓
[Firestore に保存]
  - contextualized_content: コンテキスト付きテキスト（検索用）
  - original_content: 元チャンク（LLMへの回答生成用）
```

**Step 2: コスト最適化 — Vertex AI Context Caching の活用**
- Gemini 2.5 以降のモデルでは、キャッシュされたトークンは**通常の入力トークン料金の10%**で処理可能
- 1ドキュメント（8,000トークン）を10チャンクに分割する場合、ドキュメント全文は1回だけキャッシュし、各チャンクの処理では**キャッシュ済みトークン（90%割引）+ チャンク分のトークン（通常料金）**のみ
- Implicit Caching（自動キャッシュ）を使えばストレージ料金も不要

**Step 3: Firestore スキーマ拡張**
```
chunks/{chunkId}:
  content: string           # 元のチャンクテキスト
  contextualized_content: string  # コンテキスト付きテキスト（NEW）
  context_summary: string   # LLM生成の文脈説明文のみ（NEW）
  embedding: vector         # contextualized_content から生成
  document_title: string    # 既存のヘッダーインジェクション用
  ...
```

**コスト試算（参考）:**

| 項目 | 値 |
|------|-----|
| Anthropic公式の参考値 | $1.02 / 100万ドキュメントトークン |
| Gemini 2.5 Flash（Context Caching適用後の概算） | 同等〜やや安価（Flash は Claude Haiku より安価、かつキャッシュ90%割引） |
| 1,000チャンク（想定文書量）の概算 | 数ドル程度（一回限りのインデクシングコスト） |

**PoC/本番での採用判断:**

| 観点 | 判断 | 理由 |
|------|------|------|
| **PoC** | **保留（優先度: 中）** | 現在のヘッダーインジェクション + リランキングで一定の精度が出ている。まずハイブリッド検索（最優先）の実装が先。ただし実装難易度は低い（ingestパイプラインに1ステップ追加するだけ）ため、ハイブリッド検索後の次の改善候補として有力 |
| **本番** | **採用推奨** | インデクシング時の一回限りのコストで、検索時のレイテンシ増加なし。2026年時点でRAGのベストプラクティスとして定着。ヘッダーインジェクションからの移行も容易（既存フィールドを拡張するだけ） |
| **実装順序** | ハイブリッド検索 → **Contextual Retrieval** → 逆質問 | ハイブリッド検索が最優先（型番検索の弱点解消）。Contextual Retrievalは2番目に費用対効果が高い |

**補足: 2026年現在の評価**

Contextual Retrieval は2026年時点でも RAG のベストプラクティスとして広く認知されている。「ハイブリッド検索 + リランキング + コンテキスト付与」が標準的な構成とされ、Anthropic の提案した手法がそのまま業界標準化した形。ロングコンテキストウィンドウの拡大（Gemini 2.5 で100万トークン対応）により「RAG不要論」も出たが、コスト・レイテンシ・権限制御・監査性の観点から RAG は依然として必要とされ、その中で Contextual Retrieval は検索精度を上げる定番手法の位置づけ。

#### 2. マルチモーダルRAG

**概要:**

テキストだけでなく画像・図表・フローチャートもRAGの検索・回答生成に活用する手法。企業ドキュメント（部品仕様書の図面、マニュアルのフローチャート等）ではテキストだけでは情報が不完全になるため重要。

主要な2つのアプローチ:

| アプローチ | 概要 | メリット | デメリット |
|---|---|---|---|
| **A. 画像→テキスト変換** | マルチモーダルLLM（Gemini等）で画像を説明テキストに変換し、通常のテキストRAGパイプラインで処理 | 既存パイプラインに簡単に統合。テキスト埋め込みモデルがそのまま使える | 視覚的な空間関係・レイアウト情報が失われる |
| **B. マルチモーダル埋め込み** | テキストと画像を同一ベクトル空間に埋め込み、クロスモーダル検索を実現 | 視覚情報をそのまま保持 | 専用モデルが必要。Firestoreの次元数制限（2048）との整合が必要 |

**対象文書の図表含有度:**

- 部品仕様書: 図面・寸法図が含まれる可能性が高い
- マニュアル: フローチャート・操作画面のスクリーンショットが含まれる
- 現時点のテストデータはMarkdownテキストのみ → PoCでは図表対応は検証対象外

**Geminiでの実現方法:**

| モデル | 用途 | 備考 |
|--------|------|------|
| **Gemini 2.5 Flash** | 画像→説明テキスト生成 | 1024x1024画像 ≈ 1,290トークン。入力$0.30/100万トークン |
| **Gemini Embedding 2** | マルチモーダル埋め込み | 2026年3月10日 Public Preview。3,072次元（768に縮小可能）。テキスト・画像・動画・音声対応 |
| **multimodalembedding@001** | マルチモーダル埋め込み（既存） | 1,408次元。ただしテキスト入力が最大32トークンという重大な制限あり |

**推奨実装パターン（パターンA: PoC向け）:**
```
[PDF/画像] → Gemini 2.5 Flash で説明テキスト生成
  → テキストチャンクとして保存（metadataに画像URL保持）
  → text-embedding-005 でベクトル化（既存パイプラインそのまま）
  → 回答生成時: 元画像を Gemini に渡して精度向上
```

**コスト:** 画像100枚の説明生成 ≈ $0.09、画像1,000枚 ≈ $0.9（一回限り）

**PoC/本番での採用判断:**

| 観点 | 判断 | 理由 |
|------|------|------|
| **PoC** | **見送り** | 現在のテストデータはテキストのみ。対象文書に図表が含まれることが確認されてから着手すべき |
| **本番** | **採用推奨（パターンA → B段階移行）** | パターンA（画像→テキスト変換）で開始し、Gemini Embedding 2のGA後にパターンB（マルチモーダル埋め込み）へ移行 |
| **前提条件** | 対象文書に図表・画像がどの程度含まれるか顧客に確認が必要 | 図表がなければ優先度は大きく下がる |

#### 3. Metadata-aware Retrieval

**概要:**

ベクトル類似度だけでなく、メタデータ（部署、文書種別、更新日、製品カテゴリ等）を**独立したスコアリング軸**として検索ランキングに組み込む手法。「3年前の古い文書が、言語的に類似しているだけで最新の手順書より上位に表示される」問題を解決する。

主要な実装アプローチ:

| アプローチ | 説明 | 精度向上実績 |
|---|---|---|
| **Post-retrieval スコアリング** | ベクトル検索後にメタデータスコアを加算 | 実装容易、即効性あり |
| **Weighted Score Fusion** | `α × vector_score + (1-α) × metadata_score` で統合 | 精度: +12%（Unstructured.io） |
| **Unified Embedding** | テキストとメタデータを統合ベクトルに | Context@K: 33% → 63%（+30pt） |

**Firestoreでの実現方法:**

**パターンA: Post-retrieval メタデータスコアリング（推奨・即時実装可能）:**

Firestoreのベクトル検索結果を取得後、アプリケーション層でスコア調整する。

```
[ベクトル検索] top_k * 3 で多めに取得
  → [メタデータスコア加算]
      カテゴリ一致: +0.15
      更新日が90日以内: +0.10
  → [再ソート] → top_k に絞り込み
  → [リランキング]（既存）
```

既存の `retriever.py` に数十行の追加で実装可能。新しいGCPサービスやライブラリは不要。

**既存メタデータとの関係:**

| フィールド | 既存 | 用途 |
|-----------|------|------|
| `category` | ✅ あり | カテゴリ一致でスコアブースト |
| `security_level` | ✅ あり | Pre-filtering用（スコアリングではなくアクセス制御） |
| `allowed_groups` | ✅ あり | Pre-filtering用 |
| `source_file` | ✅ あり | ファイル名一致でブースト可能 |
| `updated_date` | ❌ なし | **追加推奨**。最新文書の優先に使える |
| `doc_type` | ❌ なし | 追加検討。規程/マニュアル/仕様書の区別 |

**重要な区別:** Pre-filtering（DLS）とMetadata-aware Retrievalは**併用**すべき。
- Pre-filtering: セキュリティ（二値: 見える/見えない）→ 必須のアクセス制御
- Metadata-aware: ランキング（連続値: スコア加算）→ 品質改善

**PoC/本番での採用判断:**

| 観点 | 判断 | 理由 |
|------|------|------|
| **PoC** | **採用候補（優先度: 中）** | パターンA（Post-retrieval）は既存コードへの少量追加で実装可能。ただし `updated_date` 等のメタデータ拡張が前提 |
| **本番** | **採用推奨** | 日本語企業文書では特に効果が高い（同じテンプレートで作られた文書の区別にメタデータが不可欠） |
| **既存技術との関係** | Pre-filteringの**補完**であり**代替ではない** | セキュリティはPre-filter、ランキング改善はMetadata-aware、と役割が明確に分かれる |

#### 4. Self-Query Retrieval

**概要:**

ユーザーの自然言語クエリをLLMが解析し、**セマンティック検索用のクエリ**と**構造化フィルタ条件**を自動的に分離・生成する手法。LangChainの `SelfQueryRetriever` がデファクト実装。

```
入力: 「2024年以降の人事規程を教えて」
  ↓ LLMが解析
セマンティッククエリ: 「人事規程」
構造化フィルタ: { category: "hr_policy", date_after: "2024-01-01" }
```

参考精度（ベンチマーク）:

| 手法 | Precision | Recall | F1 |
|------|-----------|--------|----|
| Basic Vector | 75% | 70% | 72.5% |
| Hybrid Retrieval | 85% | 82% | 83.5% |
| **Self-Query** | **88%** | **85%** | **86.5%** |

**Firestoreクエリとの組み合わせ:**

LangChainにFirestore用Translatorが存在しないため、自前実装が現実的:

```
[ユーザークエリ] → [Gemini でフィルタ条件をJSON生成]
  → [バリデーション（既知のフィールド名・許容値と照合）]
  → [Firestore where() を動的構築 + find_nearest()]
  → [フィルタ失敗時はフィルタなし検索にフォールバック]
```

Firestoreの `find_nearest()` と `where()` の組み合わせは対応済み（事前フィルタリング）。

**インテントルーティングとの違い:**

| 観点 | Self-Query Retrieval | Intent Routing |
|------|---------------------|----------------|
| **目的** | 1つのクエリからフィルタ条件を抽出 | クエリの種別を判定し検索戦略を切り替え |
| **作用点** | メタデータフィルタの生成 | パイプライン全体のルーティング |
| **粒度** | フィールドレベル（category, date等） | パイプラインレベル（どのDBに行くか） |
| **関係** | Intent Routingの**下流**で動作 | Self-Queryの**上流**で動作 |

**両者は排他的ではなく補完関係。** Intent Routing → 「文書検索が必要」と判定 → Self-Query → フィルタ条件生成、という2段構成が理想。

**リスク:**
- フィルタの誤生成（存在しないフィールド名、型不一致）→ バリデーション層で対処
- 過剰フィルタリング（結果0件）→ フォールバック + 段階的フィルタ緩和
- 日本語の曖昧性（「最近の」「古い」）→ 正確なフィルタ生成が困難
- Geminiの `response_schema` でJSON出力を強制することで安定性向上

**PoC/本番での採用判断:**

| 観点 | 判断 | 理由 |
|------|------|------|
| **PoC** | **保留（優先度: 低）** | Intent Routing（未実装）が上流で先に必要。メタデータスキーマの充実も前提。現状の `category` と `security_level` だけでは効果限定的 |
| **本番** | **条件付き採用** | `date`, `department`, `doc_type` 等のメタデータが充実した段階で導入。フォールバック・モニタリング必須 |
| **実装順序** | Intent Routing → Metadata-aware → **Self-Query** | 上流から順に整備するのが自然 |
