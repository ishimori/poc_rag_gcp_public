### 実装順序の設計

依存関係・難易度・効果のバランスから、以下の順序で子DDを実装する。同時着手可能なものは並行して進める。

**グループA（並行実装可能・低コスト）**: DD-019-1, DD-019-2, DD-019-3
**グループB（並行実装可能・中コスト）**: DD-019-4, DD-019-5 — グループA完了後
**グループC（順次実装・依存あり）**: DD-019-6 → DD-019-7 — グループB完了後

### 評価サイクル

DD-013で導入したLLM-as-Judgeがベースライン。

**子DD完了時**: `--limit 10` の簡易評価（1〜2分）で動作確認・明らかな問題がないか確認
**グループ完了時**: `evaluate.py` フル実行（74件、約10分）でスコアを正式記録

```
子DD実装 → 簡易評価(--limit 10) → 次の子DDへ → ... → グループ完了 → フル評価 → スコア記録
```

個別技術の貢献度はconfigのON/OFF切替で後から測定可能（DD-013-2のアブレーション再実行で扱う）。

### DD-013との関係

DD-013のPhase 2〜4（未実装テスト分離表示、品質レビュー、件数均等化）は本DDとは独立して進行する。ただし、DD-019-2/DD-019-5の実装で ambiguous/security のスコアが改善されると、DD-013 Phase 2（未実装テスト分離）の緊急度は下がる。

## 決定事項

- **各子DD完了時**: `--limit 10` 簡易評価で動作確認。フル評価はグループ完了時にまとめて実施

## 申し送り（子DD間の引き継ぎ事項）

### DD-019-1 → DD-019-7: `updated_at`（更新日）によるスコアリング

DD-019-1（#11 メタデータスコアリング）で当初予定していた更新日ボーナス軸を除外した。

**理由**: PoCでは全ソース文書を一括Ingestするため、Ingest日が全文書で同一になり日付ボーナスが機能しない。フロントマターへの手動追加はテストデータの「仕込み」となりPoCの説得力を下げる。

**本番での解法**: 文書管理システム（SharePoint、Confluence等）のAPIから更新日を取得できるため、本番環境ではこの問題は発生しない。

**PoCでのtemporal改善**: DD-019-7（#12 セルフクエリ）でクエリから「2026年版」等のフィルタ条件を自動抽出する方が、日付ボーナスよりも根本的な解法。DD-019-7ではtemporal 0%の改善を主要目標の一つとする。
