# DD-013-3: 評価実行の進捗表示

| 作成日 | 更新日 | ステータス |
|--------|--------|------------|
| 2026-03-21 | 2026-03-21 | 完了 |

## 目的

評価パイプライン（64〜74件、実行時間10分以上）の実行中に進捗を可視化する。CLI・Admin UI両方で対応する。

## 背景・課題

### 現状の問題

| 実行経路 | 仕組み | 進捗表示 |
|----------|--------|---------|
| **CLI**: `uv run python scripts/evaluate.py` | 直接実行。printで逐次出力 | ターミナルに出るが、残り時間が不明 |
| **Admin UI**: Tuning画面 → Evaluateボタン | `POST /api/admin/evaluate` → main.py `_handle_evaluate()` | レスポンスが返るまで**何も見えない**（10分以上待つ） |

### 要件

- CLI: `[12/64] FAIL exact-005 (llm: incorrect) [2:15 elapsed, ~12:30 remaining]` のような進捗表示
- Admin UI: 実行中の進捗をリアルタイムで表示（プログレスバー or ケースごとの逐次表示）

## 検討内容

### Admin UI の進捗表示方式

| 案 | 内容 | メリット | デメリット |
|----|------|---------|-----------|
| A: インメモリ進捗ポーリング | 評価中にグローバル辞書へ進捗を書き込み、UIが `GET /evaluate/status` をポーリング | **最もシンプル。既存のPOSTエンドポイントをほぼ変更不要** | ポーリング間隔分の遅延（2秒程度） |
| B: Server-Sent Events (SSE) | `/evaluate` をSSEエンドポイントに変更し、ケースごとにイベント送信 | リアルタイム | エンドポイントの応答方式を根本変更。`EventSource`はPOST非対応のためGET化 or fetch+ReadableStream必要 |
| C: バックグラウンド実行 + 結果確認 | POSTで即座に202を返し、バックグラウンドで実行。UIは結果を定期確認 | UI がブロックされない | 実装が複雑。進捗は見えない（完了/未完了のみ） |

### タイムアウトの検証

- `main.py` の `admin` 関数に `timeout_sec=540` が設定されている
- ただし `dev.sh` は **Firebase Emulatorではなく `functions-framework` で直接起動** しているため、`timeout_sec` は適用されない
- Tuning画面のIngest/Evaluateは `isProd` チェックで **ローカル環境限定** のため、Cloud Functionsのタイムアウト制約は無関係

### CLI の進捗表示

`runner.py` の `run_evaluation()` に `time.time()` でタイマーと残り時間推定を追加するだけ。シンプル。

表示フォーマット:
```
[12/64] PASS exact-005 (llm: correct, keyword: 100%) [2:15 elapsed, ~12:30 remaining]
```

## 決定事項

- **Admin UI方式 → A（インメモリポーリング）**
  - 既存の `POST /evaluate` はほぼそのまま。ループ中にグローバル辞書へ進捗を書き込む
  - `GET /evaluate/status` エンドポイントを1つ追加し、UIが2秒間隔でポーリング
  - Firestoreは不要（ローカル限定のためインメモリで十分）
  - B案（SSE）は既存エンドポイント構造を壊す。A案の方がシンプル
- **CLI → `run_evaluation()` に経過時間・残り時間推定を追加**
- **並行リクエスト処理**: `functions-framework` はスレッドプール（Gunicorn gthread or Flask threaded）で動作するため、POST実行中でもGETポーリングは別スレッドで処理される

### Phase 1: CLI進捗表示 — `runner.py` 変更

**変更箇所**: `run_evaluation()` のみ

```python
import time

def run_evaluation(cases: list[EvalCase]) -> list[EvalResult]:
    results: list[EvalResult] = []
    total = len(cases)
    start_time = time.time()
    active_count = 0  # skipped以外の実行済み件数

    for i, eval_case in enumerate(cases, 1):
        result = run_case(eval_case)

        elapsed = time.time() - start_time
        elapsed_str = _format_time(elapsed)

        if result.skipped:
            print(f"  [{i}/{total}] SKIP {result.id} ({result.skipped_reason}) [{elapsed_str} elapsed]")
        else:
            active_count += 1
            status = "PASS" if result.passed else "FAIL"
            disc = f" ⚠ {result.discrepancy}" if result.discrepancy else ""

            # 残り時間推定: activeケースの平均実行時間 × 残りケース数
            avg_per_active = elapsed / active_count if active_count > 0 else 0
            remaining_cases = total - i
            est_remaining = avg_per_active * remaining_cases
            remaining_str = f", ~{_format_time(est_remaining)} remaining" if active_count >= 2 else ""

            print(f"  [{i}/{total}] {status} {result.id} (llm: {result.llm_label}, keyword: {result.keyword_score * 100:.0f}%){disc} [{elapsed_str} elapsed{remaining_str}]")

        results.append(result)
    ...
```

**設計ポイント**:
- **残り時間推定**: skippedケースは一瞬で終わるため、推定の平均計算には含めない。ただし残りケース数には含める（skippedケースも並んでいるため）
- **推定表示は2件目以降**: 1件だけではサンプル不足で推定が不正確。`active_count >= 2` で表示開始
- **`_format_time()`**: 秒数を `M:SS` 形式に変換するヘルパー

**問題点**: 残りケースのうち何件がskippedになるかは事前に分からない。しかしskippedは一瞬で終わるので、推定が多少長めに出るだけで実害なし。

### Phase 2: Admin UI進捗表示 — 3層の変更

#### 2-1. バックエンド: `main.py`

**グローバル進捗辞書**:
```python
_eval_progress: dict = {
    "running": False,
    "current": 0,
    "total": 0,
    "current_id": "",
    "elapsed": 0,
    "estimated_remaining": 0,
    "results": [],  # [{"id": "exact-001", "status": "PASS", "llm_label": "correct"}, ...]
}
```

**`_handle_evaluate()` の変更**:
- ループ前に `_eval_progress["running"] = True, total = len(cases)` をセット
- 各ケース完了時に `current`, `current_id`, `elapsed`, `estimated_remaining`, `results` を更新
- ループ後に `running = False` に戻す
- 残り時間推定ロジックはPhase 1の `runner.py` と同じ（ただし `runner.py` を直接呼ぶので、**コールバックで進捗を受け取る**か、**`runner.py` にフック機構を追加**する）

**進捗更新の方法 — 2案**:

| 案 | 内容 | メリット | デメリット |
|----|------|---------|-----------|
| コールバック | `run_evaluation(cases, on_progress=callback)` | runner.pyがmain.pyのグローバル変数を知らなくてよい | 引数追加 |
| 直接import | `runner.py` から `main.py` の辞書をimport | 変更不要 | 循環import、結合度が高い |

→ **コールバック方式**を採用。`run_evaluation()` にオプショナルな `on_progress` コールバックを追加。

```python
# runner.py
from typing import Callable

ProgressCallback = Callable[[int, int, EvalResult], None]  # (current, total, result)

def run_evaluation(cases: list[EvalCase], on_progress: ProgressCallback | None = None) -> list[EvalResult]:
    ...
    for i, eval_case in enumerate(cases, 1):
        result = run_case(eval_case)
        ...
        if on_progress:
            on_progress(i, total, result)
        ...
```

```python
# main.py
def _handle_evaluate(req):
    ...
    def _on_progress(current, total, result):
        _eval_progress.update({
            "current": current,
            "total": total,
            "current_id": result.id,
            "elapsed": ...,
            "results": ...,
        })

    results = run_evaluation(cases, on_progress=_on_progress)
    _eval_progress["running"] = False
    ...
```

**`GET /evaluate/status` エンドポイント**:
```python
def _handle_evaluate_status(req):
    return _json_response(_eval_progress)
```

ルーティングに追加:
```python
if path == "/evaluate/status" and method == "GET":
    return _handle_evaluate_status(req)
```

#### 2-2. API層: `ui/src/admin/api.ts`

```typescript
export interface EvalProgress {
  running: boolean
  current: number
  total: number
  current_id: string
  elapsed: number
  estimated_remaining: number
  results: { id: string; status: string; llm_label: string }[]
}

export function getEvalStatus(): Promise<EvalProgress> {
  return request('/evaluate/status')
}
```

#### 2-3. フロントエンド: `ui/src/admin/Tuning.tsx`

**ポーリングのライフサイクル**:
1. `handleEvaluate()` / `handleRetune()` でPOST開始
2. 同時に `setInterval(getEvalStatus, 2000)` でポーリング開始
3. `running === false` になったらポーリング停止、clearInterval
4. コンポーネントアンマウント時もcleanup

**UI表示**:
```
評価中... [12/N] exact-005 (llm: correct)
[=========>          ] 18.8%  |  2:15 elapsed, ~12:30 remaining
```
- プログレスバー: `current / total` で幅を計算
- 現在のケースID + 結果
- 経過時間 + 残り時間

**状態管理の追加**:
```typescript
const [evalProgress, setEvalProgress] = useState<EvalProgress | null>(null)
```

既存の `evalStatus === 'loading'` の間だけポーリングを行う。`useEffect` で `evalStatus` を監視。
